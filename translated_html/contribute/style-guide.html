Espressif IoT-ის შემუშავების ჩარჩოს სტილის სახელმძღვანელო¶

ამ სახელმძღვანელოს შესახებ¶

ამ სტილის სახელმძღვანელოს მიზანია, წაახალისოს ESP-IDF ის ფარგლებში კოდირების გავრცელებული პრაქტიკის გამოყენება.

სტილის სახელმძღვანელო არის წესების ერთობლიობა, რომელიც მიზნად ისახავს წაკითხვადი, მხარდაჭერისა და საიმედო კოდის შექმნას. კოდის ბაზაში ერთნაირად იერსახის მქონე კოდის დაწერით, ჩვენ ვეხმარებით სხვებს კოდის წაკითხვასა და გაგებაში. სივრცეებისა და ახალი ხაზებისთვის იგივე კონვენციების გამოყენებით, ჩვენ ვამცირებთ იმის შანსებს, რომ მომავალი ცვლილებები გამოიწვევს უზარმაზარ წაუკითხავ განსხვავებებს. მოდულის სტრუქტურის საერთო ნიმუშების დაცვით და ენობრივი მახასიათებლების თანმიმდევრული გამოყენებით, ჩვენ ვეხმარებით სხვებს კოდის ქცევის გაგებაში.

ჩვენ ვცდილობთ, წესები საკმარისად მარტივი იყოს, რაც იმას ნიშნავს, რომ ისინი ყველა პოტენციურ შემთხვევას ვერ მოიცავენ. ზოგიერთ შემთხვევაში, წაკითხვადობის, მდგრადობის ან სიმტკიცის მისაღწევად, ამ მარტივი წესების დარღვევაა საჭირო.

ESP-IDF ში გამოყენებული მესამე მხარის კოდის მოდიფიკაციისას, მიჰყევით კონკრეტული პროექტის დაწერის წესს. ეს დაგეხმარებათ სასარგებლო ცვლილებების შეთავაზებაში ზედა დინამიკურ პროექტში გაერთიანებისთვის.

C კოდის ფორმატირება¶

ჩაღრმავება¶

ჩაღრმავების თითოეული დონისთვის გამოიყენეთ 4 ინტერვალი. ჩაღრმავებისთვის Tab-ების გამოყენება არ გამოიყენოთ. დააკონფიგურირეთ რედაქტორი ისე, რომ Tab ღილაკზე ყოველ ჯერზე დაჭერისას 4 ინტერვალი გამოჩნდეს.

ვერტიკალური სივრცე¶

ფუნქციებს შორის ერთი ცარიელი ხაზი მოათავსეთ. ფუნქცია არ დაიწყოთ ან დაასრულოთ ცარიელი ხაზით.

void function1()
{
    do_one_thing();
    do_another_thing();
                                // INCORRECT, don't place empty line here
}
                                // place empty line here
void function2()
{
                                // INCORRECT, don't use an empty line here
    int var = 0;
    while (var < SOME_CONSTANT) {
        do_stuff(&var);
    }
}

მაქსიმალური ხაზის სიგრძე 120 სიმბოლოა, იმ პირობით, რომ ეს სერიოზულად არ იმოქმედებს წაკითხვადობაზე.

ჰორიზონტალური სივრცე¶

პირობითი და ციკლის საკვანძო სიტყვების შემდეგ ყოველთვის დაამატეთ ერთი ინტერვალი:

if (condition) {    // correct
    // ...
}

switch (n) {        // correct
    case 0:
        // ...
}

for(int i = 0; i < CONST; ++i) {    // INCORRECT
    // ...
}

ორობითი ოპერატორების გარშემო დაამატეთ ერთიანი ინტერვალი. უნარული ოპერატორებისთვის ინტერვალი საჭირო არ არის. გამრავლებისა და გაყოფის ოპერატორების გარშემო ინტერვალის გამოტოვება დასაშვებია:

const int y = y0 + (x - x0) * (y1 - y0) / (x1 - x0);    // correct

const int y = y0 + (x - x0)*(y1 - y0)/(x1 - x0);        // also okay

int y_cur = -y;                                         // correct
++y_cur;

const int y = y0+(x-x0)*(y1-y0)/(x1-x0);                // INCORRECT

ირგვლივ სივრცე არ არის საჭირო . და -> ოპერატორები.

ზოგჯერ ხაზში ჰორიზონტალური სივრცის დამატება კოდის წაკითხვადობას უწყობს ხელს. მაგალითად, ფუნქციის არგუმენტების გასასწორებლად შეგიძლიათ სივრცის დამატება:

gpio_matrix_in(PIN_CAM_D6,   I2S0I_DATA_IN14_IDX, false);
gpio_matrix_in(PIN_CAM_D7,   I2S0I_DATA_IN15_IDX, false);
gpio_matrix_in(PIN_CAM_HREF, I2S0I_H_ENABLE_IDX,  false);
gpio_matrix_in(PIN_CAM_PCLK, I2S0I_DATA_IN15_IDX, false);

თუმცა, გაითვალისწინეთ, რომ თუ ვინმე ახალ სტრიქონს დაამატებს უფრო გრძელი იდენტიფიკატორით, როგორც პირველი არგუმენტი (მაგ. PIN_CAM_VSYNC), ის არ მოერგება. ამიტომ, სხვა ხაზების ხელახლა გასწორება იქნება საჭირო, რაც კომიტს უაზრო ცვლილებებს დაუმატებს.

ამიტომ, ჰორიზონტალური გასწორება ზომიერად გამოიყენეთ, განსაკუთრებით თუ მოგვიანებით სიაში ახალი ხაზების დამატებას ელოდებით.

ჰორიზონტალური გასწორებისთვის არასოდეს გამოიყენოთ TAB სიმბოლოები.

ხაზის ბოლოს არასდროს დაამატოთ ცარიელი ადგილები.

ბრეკეტები¶

  • ფუნქციის განმარტებას ცალკე ხაზზე უნდა ჰქონდეს ფრჩხილი:

    // This is correct:
void function(int arg)
{

}

// NOT like this:
void function(int arg) {

}

  • ფუნქციის ფარგლებში, გახსნის ფრჩხილი მოათავსეთ პირობით და ციკლის ოპერატორებთან ერთ ხაზზე:

    if (condition) {
    do_one();
} else if (other_condition) {
    do_two();
}

კომენტარები¶

გამოყენება // ერთსტრიქონიანი კომენტარებისთვის. მრავალსტრიქონიანი კომენტარებისთვის დასაშვებია ორივეს გამოყენება // თითოეულ ხაზზე ან /* */ ბლოკი.

მიუხედავად იმისა, რომ ფორმატირებასთან პირდაპირ კავშირში არ არის, აქ მოცემულია რამდენიმე შენიშვნა კომენტარების ეფექტურად გამოყენების შესახებ.

  • არ გამოიყენოთ ცალკეული კომენტარები ზოგიერთი ფუნქციონალის გამოსართავად:

    void init_something()
{
    setup_dma();
    // load_resources();                // WHY is this thing commented, asks the reader?
    start_timer();
}

  • თუ კოდის ნაწილი აღარ არის საჭირო, მთლიანად წაშალეთ იგი. თუ დაგჭირდებათ, ყოველთვის შეგიძლიათ მისი მოძებნა ამ ფაილის git ისტორიაში. თუ დროებითი მიზეზების გამო გამორთავთ რაიმე გამოძახებას, მომავალში მისი აღდგენის განზრახვით, დაამატეთ ახსნა მიმდებარე ხაზზე:

    void init_something()
{
    setup_dma();
    // TODO: we should load resources here, but loader is not fully integrated yet.
    // load_resources();
    start_timer();
}

  • იგივე ეხება #if 0 ... #endif ბლოკები. კოდის ბლოკის სრულად წაშლა, თუ ის არ გამოიყენება. წინააღმდეგ შემთხვევაში, დაამატეთ კომენტარი, რომელიც ხსნის, თუ რატომ არის ბლოკი გამორთული. არ გამოიყენოთ #if 0 ... #endif ან კომენტარები კოდის ფრაგმენტების შესანახად, რომლებიც შეიძლება მომავალში დაგჭირდეთ.

  • ნუ დაამატებთ უმნიშვნელო კომენტარებს ავტორობისა და ცვლილების თარიღის შესახებ. git-ის გამოყენებით ყოველთვის შეგიძლიათ გაარკვიოთ, ვინ შეცვალა მოცემული ხაზი. მაგალითად, ეს კომენტარი კოდს აურზაურს მატებს რაიმე სასარგებლო ინფორმაციის დამატების გარეშე:

    void init_something()
{
    setup_dma();
    // XXX add 2016-09-01
    init_dma_list();
    fill_dma_item(0);
    // end XXX add
    start_timer();
}

სტრიქონის დაბოლოებები¶

კომიტებმა უნდა შეიცავდნენ მხოლოდ LF (Unix სტილის) დაბოლოების მქონე ფაილებს.

Windows-ის მომხმარებლებს შეუძლიათ git-ის კონფიგურაცია CRLF (Windows-ის სტილი) დაბოლოებების ლოკალურად შესამოწმებლად და LF დაბოლოებების დამადასტურებლად, შემდეგი პარამეტრის დაყენებით: core.autocrlf გარემოში. Github has a document about setting this option <github-line-endings>თუმცა, რადგან MSYS2 იყენებს Unix სტილის სტრიქონების დაბოლოებებს, ხშირად უფრო ადვილია ტექსტური რედაქტორის კონფიგურაცია ისე, რომ ESP-IDF საწყისი ფაილების რედაქტირებისას გამოიყენოს LF (Unix სტილი) დაბოლოებები.

თუ თქვენს ფილიალში შემთხვევით გაქვთ რამდენიმე კომიტი, რომელიც LF დაბოლოებებს დაამატებს, შეგიძლიათ მათი Unix-ად გარდაქმნა MSYS2-ში ან Unix ტერმინალში ამ ბრძანების შესრულებით (შეცვალეთ დირექტორია IDF სამუშაო დირექტორიაში და წინასწარ შეამოწმეთ, რომ ამჟამად სწორი განშტოებაა გამორთული):

git rebase --exec 'git diff-tree --no-commit-id --name-only -r HEAD | xargs dos2unix && git commit -a --amend --no-edit --allow-empty' master

(გაითვალისწინეთ, რომ ეს ხაზი ხელახლა ბაზირდება master-ზე, შეცვალეთ ტოტის სახელი ბოლოს, რათა ხელახლა ბაზირდეს სხვა ტოტზე.)

ერთი commit-ის განახლებისთვის შესაძლებელია მისი გაშვება dos2unix FILENAME და შემდეგ გაიქეცი git commit --amend

თქვენი კოდის ფორმატირება¶

შეგიძლიათ გამოიყენოთ astyle პროგრამა, რომელიც თქვენს კოდს ზემოთ მოცემული რეკომენდაციების შესაბამისად დააფორმატებს.

თუ ფაილს ნულიდან წერთ ან სრულ გადაწერას ახორციელებთ, თავისუფლად შეგიძლიათ მთელი ფაილის ხელახლა ფორმატირება. თუ ფაილის მცირე ნაწილს ცვლით, ნუ გადააფორმატებთ იმ კოდს, რომელიც არ შეგიცვლიათ. ეს სხვებს დაეხმარება თქვენი ცვლილებების გადახედვისას.

ფაილის ხელახლა ფორმატირებისთვის, გაუშვით:

tools/format.sh components/my_component/file.c

C++ კოდის ფორმატირება¶

C-ს შემთხვევაში იგივე წესები მოქმედებს. იმ შემთხვევაში, თუ ისინი საკმარისი არ არის, გამოიყენეთ შემდეგი წესები.

ფაილის დასახელება¶

C++ სათაურის ფაილებს აქვთ გაფართოება .hppC++ საწყისი ფაილებს აქვთ გაფართოება .cppეს უკანასკნელი მნიშვნელოვანია კომპილაცია r-ისთვის, რათა განასხვავოს ისინი ჩვეულებრივი C წყაროს ფაილებისგან.

დასახელება¶

  • კლასი და სტრუქტურა სახელები დაიწერება, CamelCase დიდი ასოთი იწყება. წევრი ცვლადები და მეთოდები უნდა იყოს snake_case.
  • სახელთა სივრცეები უფრო დაბალ დონეზე იქნება snake_case.
  • შაბლონები მითითებულია ფუნქციის დეკლარაციის ზემოთ მოცემულ ხაზში.
  • ობიექტზე ორიენტირებული პროგრამირების თვალსაზრისით ინტერფეისები უნდა დასახელდეს სუფიქსის გარეშე. ...Interfaceმოგვიანებით, ეს აადვილებს ინტერფეისების ამოღებას ჩვეულებრივი კლასებიდან და პირიქით, კრიტიკული ცვლილების შეტანის გარეშე.

წევრების თანმიმდევრობა კლასებში¶

პრიორიტეტის მიხედვით:

  • ჯერ საჯარო წევრები, შემდეგ დაცული და შემდეგ კერძო წევრები მოათავსეთ. გამოტოვეთ საჯარო, დაცული ან კერძო სექციები წევრების გარეშე.
  • ჯერ ჩავსვათ კონსტრუქტორები/დესტრუქტორები, შემდეგ წევრი ფუნქციები და შემდეგ წევრი ცვლადები.

მაგალითად:

class ForExample {
public:
    // first constructors, then default constructor, then destructor
    ForExample(double example_factor_arg);
    ForExample();
    ~ForExample();

    // then remaining pubic methods
    set_example_factor(double example_factor_arg);

    // then public member variables
    uint32_t public_data_member;

private:
    // first private methods
    void internal_method();

    // then private member variables
    double example_factor;
};

ინტერვალი¶

  • სახელთა სივრცეებში არ გააკეთოთ შეწევა.
  • დადეთ public, protected და private ეტიკეტები იმავე ჩაღრმავების დონეზე, როგორც შესაბამისი class ეტიკეტი.

მარტივი მაგალითი¶

// file spaceship.h
#ifndef SPACESHIP_H_
#define SPACESHIP_H_
#include <cstdlib>

namespace spaceships {

class SpaceShip {
public:
    SpaceShip(size_t crew);
    size_t get_crew_size() const;

private:
    const size_t crew;
};

class SpaceShuttle : public SpaceShip {
public:
    SpaceShuttle();
};

class Sojuz : public SpaceShip {
public:
    Sojuz();
};

template <typename T>
class CargoShip {
public:
    CargoShip(const T &cargo);

private:
    T cargo;
};

} // namespace spaceships

#endif // SPACESHIP_H_

// file spaceship.cpp
#include "spaceship.h"

namespace spaceships {

// Putting the curly braces in the same line for constructors is OK if it only initializes
// values in the initializer list
SpaceShip::SpaceShip(size_t crew) : crew(crew) { }

size_t SpaceShip::get_crew_size() const
{
    return crew;
}

SpaceShuttle::SpaceShuttle() : SpaceShip(7)
{
    // doing further initialization
}

Sojuz::Sojuz() : SpaceShip(3)
{
    // doing further initialization
}

template <typename T>
CargoShip<T>::CargoShip(const T &cargo) : cargo(cargo) { }

} // namespace spaceships

CMake კოდის სტილი¶

  • შეწევა ოთხი ინტერვალით.
  • ხაზის მაქსიმალური სიგრძე 120 სიმბოლოა. ხაზების გაყოფისას, შეძლებისდაგვარად, შეეცადეთ ყურადღება გაამახვილოთ წაკითხვადობაზე (მაგალითად, საკვანძო სიტყვების/არგუმენტების წყვილების ცალკეულ ხაზებზე დაწყვილებით).
  • ფრჩხილებში არაფერი ჩასვათ შემდეგ endforeach(), endif()და ა.შ.
  • გამოიყენეთ პატარა ასოები ()with_underscores) ბრძანების, ფუნქციისა და მაკროების სახელწოდებებისთვის.
  • ლოკალურად განსაზღვრული ცვლადებისთვის გამოიყენეთ პატარა ასოები (with_underscores).
  • გლობალურად მასშტაბური ცვლადებისთვის გამოიყენეთ დიდი ასოები (WITH_UNDERSCORES).
  • წინააღმდეგ შემთხვევაში, დაიცავით ნაგულისხმევი პარამეტრები cmake-lint პროექტი.

პროექტის კოდის სტილის კონფიგურაცია EditorConfig-ის გამოყენებით¶

EditorConfig ეხმარება დეველოპერებს სხვადასხვა რედაქტორებსა და IDE-ებს შორის კოდირების თანმიმდევრული სტილის განსაზღვრასა და შენარჩუნებაში. EditorConfig პროექტი შედგება კოდირების სტილების განსაზღვრის ფაილის ფორმატისა და ტექსტური რედაქტორის დანამატების კოლექციისგან, რომლებიც რედაქტორებს საშუალებას აძლევს წაიკითხონ ფაილის ფორმატი და დაიცვან განსაზღვრული სტილები. EditorConfig ფაილები ადვილად იკითხება და კარგად მუშაობს ვერსიის კონტროლის სისტემებთან.

დამატებითი ინფორმაციისთვის იხილეთ რედაქტორის კონფიგურაცია ვებსაიტი.

კოდის დოკუმენტირება¶

გთხოვთ, იხილოთ სახელმძღვანელო აქ: კოდის დოკუმენტირება.

  • ნებისმიერი ცვლადი ან ფუნქცია, რომელიც გამოიყენება მხოლოდ ერთ წყაროს ფაილში, უნდა გამოცხადდეს. static.
  • საჯარო სახელებს (არასტატიკური ცვლადები და ფუნქციები) უნდა ჰქონდეს სახელთა სივრცე per- კომპონენტი ან per unit პრეფიქსით, რათა თავიდან იქნას აცილებული სახელთა კოლიზიები. მაგ. esp_vfs_register() ან esp_console_run()პრეფიქსის დაწყება esp_ Espressif-ის სპეციფიკური სახელებისთვის გამოყენება არასავალდებულოა, მაგრამ უნდა შეესაბამებოდეს იმავე კომპონენტი არსებულ სხვა სახელებს19.

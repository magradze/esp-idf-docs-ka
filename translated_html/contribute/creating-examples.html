მაგალითების შექმნა¶

[中文]

თითოეული ESP-IDF მაგალითი წარმოადგენს სრულ პროექტს, რომლის კოდის კოპირება და ადაპტირება სხვას შეუძლია საკუთარი პრობლემის გადასაჭრელად. მაგალითები უნდა აჩვენებდეს ESP-IDF ფუნქციონალურობას, ამ მიზნის გათვალისწინებით.

სტრუქტურა¶

  • ის main დირექტორია უნდა შეიცავდეს წყაროს ფაილს სახელწოდებით (something)_example_main.c ძირითადი ფუნქციონალურობით.
  • თუ მაგალითს დამატებითი ფუნქციონალობა აქვს, ლოგიკურად გაყავით იგი ცალკეულ C ან C++ წყაროს ფაილებად ქვემოთ. main და შესაბამისი სათაურის ფაილი იმავე დირექტორიაში მოათავსეთ.
  • თუ მაგალითს ბევრი დამატებითი ფუნქციონალი აქვს, განიხილეთ დამატება components მაგალითის პროექტის დირექტორიაში და შექმენით მაგალითისთვის სპეციფიკური კომპონენტი ფაილები ბიბლიოთეკის ფუნქციონალურობით. ეს მხოლოდ იმ შემთხვევაში გააკეთეთ, თუ კომპონენტი ფაილები კონკრეტული მაგალითისთვისაა, თუ ისინი ზოგადი ან საერთო ფუნქციონალურობაა, მაშინ ისინი თავად ESP-IDF ს უნდა დაემატოს.
  • მაგალითს უნდა ჰქონდეს README.md ფაილი. გამოიყენეთ შაბლონის მაგალითი README და მოარგეთ ის თქვენს კონკრეტულ მაგალითს.
  • მაგალითებს უნდა ჰქონდეთ example_test.py ფაილი ავტომატური მაგალითის ტესტის გასაშვებად. თუ წარადგენთ GitHub Pull Request-ს, რომელიც შეიცავს მაგალითს, დასაშვებია, რომ თავდაპირველად ეს ფაილი არ ჩართოთ. დეტალების განხილვა შესაძლებელია, როგორც ნაწილი. გაწევის მოთხოვნა.

ზოგადი მითითებები¶

მაგალითის კოდი უნდა მიჰყვებოდეს შემდეგს Espressif IoT-ის შემუშავების ჩარჩოს სტილის სახელმძღვანელო.

საკონტროლო სია¶

ახალი მაგალითის წარდგენამდე გასათვალისწინებელი საკონტროლო სია:

  • პროექტის სახელის მაგალითი (in Makefile და README.md) იყენებს სიტყვას „მაგალითი“. „დემო“, „ტესტი“ ან მსგავსი სიტყვების ნაცვლად გამოიყენეთ „მაგალითი“.
  • მაგალითი ერთ განსხვავებულ ფუნქციას ასრულებს. თუ მაგალითი ერთდროულად ერთზე მეტ ფუნქციას ასრულებს, გაყავით ის ორ ან მეტ მაგალითად.
  • მაგალითს აქვს README.md ფაილი, რომელიც მსგავსია შაბლონის მაგალითი README .
  • მაგალითში ფუნქციები და ცვლადები დასახელებულია შემდეგი ფორმულის მიხედვით: სტილის სახელმძღვანელოს სახელდების განყოფილება. (არასტატიკური სახელებისთვის, რომლებიც მხოლოდ მაგალითის წყაროს ფაილებისთვისაა დამახასიათებელი, შეგიძლიათ გამოიყენოთ example ან მსგავსი რამ, როგორც პრეფიქსი.)
  • მაგალითში მოცემული ყველა კოდი კარგად არის სტრუქტურირებული და კომენტარირებული.
  • მაგალითიდან ამოღებულია ნებისმიერი არასაჭირო კოდი (ძველი გამართვა ging ჟურნალები, კომენტარისგან გამოტოვებული კოდი და ა.შ.).
  • მაგალითში მოცემული პარამეტრები (მაგალითად, ქსელის სახელები, მისამართები და ა.შ.) არ არის კოდირებული. თუ შესაძლებელია, გამოიყენეთ კონფიგურაციის ელემენტები ან სხვაგვარად გამოაცხადეთ მაკროები ან მუდმივები.
  • კონფიგურაციის ელემენტები მოცემულია ა-ში KConfig.projbuild ფაილი სახელწოდებით „კონფიგურაციის მაგალითი“. იხილეთ არსებული პროექტების მაგალითები, რომ ნახოთ, თუ როგორ კეთდება ეს.
  • ყველა ორიგინალურ მაგალითის კოდს აქვს ლიცენზიის სათაური, სადაც წერია, რომ ის „საჯარო დომენშია / CC0“ და გარანტიის უარყოფის პუნქტი. ალტერნატიულად, მაგალითი ლიცენზირებულია Apache License 2.0-ით. იხილეთ არსებული მაგალითები სათაურების ადაპტაციისთვის.
  • ნებისმიერ ადაპტირებულ ან მესამე მხარის სამაგალითო კოდს აქვს ორიგინალი ლიცენზიის სათაური. ეს კოდი უნდა იყოს ლიცენზირებული Apache License 2.0-თან თავსებადი.

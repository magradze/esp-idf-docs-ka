მოვლენების დამუშავება¶

რამდენიმე ESP-IDF კომპონენტი s გამოყენება ღონისძიებები აპლიკაციის ინფორმირება მდგომარეობის ცვლილებების შესახებ, როგორიცაა კავშირი ან გათიშვა. ეს დოკუმენტი იძლევა ამ მოვლენების მექანიზმების მიმოხილვას.

Wi-Fi , Ethernet და IP მოვლენები¶

-ის დანერგვამდე esp_event ბიბლიოთეკა, Wi-Fi დრაივერის, Ethernet დრაივერის და TCP/IP დასტის მოვლენები გაიგზავნა ე.წ. მემკვიდრეობითი მოვლენების ციკლიშემდეგ ნაწილებში ახსნილია თითოეული მეთოდი.

esp_event ბიბლიოთეკის მოვლენების ციკლი¶

esp_event ბიბლიოთეკა შექმნილია ESP-IDF ში მოვლენების დამუშავების მიზნებისთვის მემკვიდრეობითი მოვლენების ციკლის ჩასანაცვლებლად. მემკვიდრეობითი მოვლენების ციკლში, ყველა შესაძლო მოვლენის ტიპი და მოვლენის მონაცემთა სტრუქტურა უნდა განისაზღვროს system_event_id_t ჩამოთვლა და system_event_info_t union-ს, რაც შეუძლებელს ხდიდა მორგებული მოვლენების გაგზავნას მოვლენების ციკლში და მოვლენების ციკლის გამოყენებას სხვა სახის მოვლენებისთვის (მაგ. Mesh). მემკვიდრეობით მიღებული მოვლენების ციკლს ასევე მხოლოდ ერთი მოვლენების დამმუშავებლის ფუნქცია უჭერდა მხარს, ამიტომ აპლიკაცია კომპონენტი s თავად ვერ ამუშავებდა Wi-Fi ან IP მოვლენების ნაწილს და აპლიკაციას სჭირდებოდა ამ მოვლენების გადამისამართება მისი მოვლენების დამმუშავებლის ფუნქციიდან.

იხილეთ esp_event ბიბლიოთეკის API მითითება ამ ბიბლიოთეკის გამოყენების შესახებ ზოგადი ინფორმაციისთვის. Wi-Fi , Ethernet და IP მოვლენები იგზავნება ნაგულისხმევი მოვლენების ციკლი ამ ბიბლიოთეკის მიერ მოწოდებული.

მემკვიდრეობითი მოვლენების ციკლი¶

ამ მოვლენის ციკლის იმპლემენტაცია იწყება გამოყენებით esp_event_loop_init() ფუნქცია. აპლიკაცია, როგორც წესი, უზრუნველყოფს მოვლენების დამმუშავებელი, ფუნქცია შემდეგი ხელმოწერით:

esp_err_t event_handler(void *ctx, system_event_t *event)
{
}

როგორც მოვლენის დამმუშავებლის ფუნქციის მიმთითებელი, ასევე თვითნებური კონტექსტის მიმთითებელი გადაეცემა esp_event_loop_init().

როდესაც Wi-Fi , Ethernet ან IP სტეკი წარმოქმნის მოვლენას, ეს მოვლენა იგზავნება მაღალი პრიორიტეტის მქონე event დავალება რიგის მეშვეობით. აპლიკაციის მიერ მოწოდებული მოვლენების დამმუშავებელი ფუნქცია გამოიძახება ამ დავალების კონტექსტში. მოვლენების დავალებების დასტის ზომისა და მოვლენების რიგის ზომის კორექტირება შესაძლებელია გამოყენებით CONFIG_ESP_SYSTEM_EVENT_TASK_STACK_SIZE და CONFIG_ESP_SYSTEM_EVENT_QUEUE_SIZE პარამეტრები, შესაბამისად.

მოვლენების დამმუშავებელი იღებს მოვლენის სტრუქტურის მაჩვენებელს (system_event_t), რომელიც აღწერს მიმდინარე მოვლენას. ეს სტრუქტურა მიჰყვება მონიშნული კავშირი ნიმუში: event_id წევრი მიუთითებს მოვლენის ტიპზე და event_info წევრი არის აღწერილობის სტრუქტურების გაერთიანება. აპლიკაციის მოვლენების დამმუშავებელი, როგორც წესი, გამოიყენებს switch(event->event_id) სხვადასხვა სახის ღონისძიებების სამართავად.

თუ აპლიკაციის მოვლენების დამმუშავებელს სჭირდება მოვლენის სხვა დავალებაზე გადაცემა, მნიშვნელოვანია აღინიშნოს, რომ მოვლენის დამმუშავებლისთვის გადაცემული მოვლენის მაჩვენებელი დროებითი სტრუქტურის მაჩვენებელია. მოვლენის სხვა დავალებაზე გადასაცემად, აპლიკაციამ უნდა შექმნას მთელი სტრუქტურის ასლი.

Mesh ღონისძიებები¶

ESP-MESH იყენებს მსგავს სისტემას მემკვიდრეობითი მოვლენების ციკლი ღონისძიებების აპლიკაციაში გადასაცემად. იხილეთ სისტემის მოვლენები დეტალებისთვის.

Bluetooth ღონისძიებები¶

Bluetooth დასტის სხვადასხვა მოდული აპლიკაციებს მოვლენებს სპეციალური უკუკავშირის ფუნქციების მეშვეობით აწვდის. უკუკავშირის ფუნქციები იღებს მოვლენის ტიპს (ჩამოთვლილი მნიშვნელობა) და მოვლენის მონაცემებს (სტრუქტურების გაერთიანება თითოეული მოვლენის ტიპისთვის). ქვემოთ მოცემულ სიაში მოცემულია API რეგისტრაციის სახელი, მოვლენის ჩამოთვლის ტიპი და მოვლენის პარამეტრების ტიპი.

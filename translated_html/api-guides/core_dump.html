ESP32 ბირთვის ნაგავსაყრელი¶

მიმოხილვა¶

ESP-IDF უზრუნველყოფს პროგრამული უზრუნველყოფის გამოუსწორებელი შეცდომების ბირთვის დემპების გენერირების მხარდაჭერას. ეს სასარგებლო ტექნიკა საშუალებას იძლევა პროგრამული უზრუნველყოფის მდგომარეობის შემდგომი ანალიზის ჩატარების დროს. ავარიის დროს სისტემა პანიკურ მდგომარეობაში გადადის, ბეჭდავს გარკვეულ ინფორმაციას და ჩერდება ან გადატვირთვა ხდება კონფიგურაციის მიხედვით. მომხმარებელს შეუძლია აირჩიოს ბირთვის დემპის გენერირება, რათა მოგვიანებით კომპიუტერზე გააანალიზოს ავარიის მიზეზი. ბირთვის დემპი შეიცავს სისტემაში არსებული ყველა დავალების სნეპშოტს ავარიის დროს. სნეპშოტები მოიცავს დავალებების მართვის ბლოკებს (TCB) და სტეკებს. ამგვარად, შესაძლებელია გაირკვეს, რომელმა დავალებამ, რომელმა ინსტრუქციამ (კოდის სტრიქონმა) და ამ დავალების რომელმა გამოძახების სტეკმა გამოიწვია ავარია. ESP-IDF უზრუნველყოფს სპეციალურ სკრიპტს. espcoredump.py რათა დაეხმაროს მომხმარებლებს ბირთვის დემპების მოძიებასა და ანალიზში. ეს ინსტრუმენტი ბირთვის დემპების ანალიზისთვის ორ ბრძანებას იძლევა:

  • info_corefile - ბეჭდავს გაფუჭებული დავალებების რეგისტრებს, გამოძახების სტეკს, სისტემაში არსებული დავალებების სიას, მეხსიერების რეგიონებს და ბირთვის დამპში შენახული მეხსიერების შინაარსს (TCB და სტეკები)
  • dbg_corefile - ქმნის core dump ELF ფაილს და ამ ფაილით გაუშვებს GDB გამართვა სესიას. მომხმარებელს შეუძლია ხელით შეამოწმოს მეხსიერება, ცვლადები და დავალებების მდგომარეობები. გაითვალისწინეთ, რომ რადგან მთელი მეხსიერება არ ინახება core dump-ში, მხოლოდ დასტაზე გამოყოფილი ცვლადების მნიშვნელობები იქნება მნიშვნელოვანი.

კონფიგურაცია¶

არსებობს ბირთვის დამპთან დაკავშირებული კონფიგურაციის რამდენიმე ვარიანტი, რომელთა არჩევა მომხმარებელს პროექტის კონფიგურაცია მენიუში შეუძლია (idf.py menuconfig).

  1. Core dump მონაცემების დანიშნულება (Components -> Core dump -> Data destination):
  • შეინახეთ ძირითადი ფაილი Flash-ში (Flash)
  • ბეჭდვა core demp UART-ზე (UART)
  • ძირითადი მონაცემების გენერირების გამორთვა (არცერთი)
  1. Core dump მონაცემთა ფორმატი (Components -> Core dump -> Core dump data format):
  • ELF ფორმატი (core dump-ისთვის შესასრულებელი და დასაკავშირებელი ფორმატის ფაილი)
  • ორობითი ფორმატი (ძირითადი ორობითი ფორმატი ძირითადი დამპისთვის)

ELF ფორმატი შეიცავს გაფართოებულ ფუნქციებს და საშუალებას იძლევა შეინახოთ მეტი ინფორმაცია გაფუჭებული ამოცანებისა და გაფუჭებული პროგრამული უზრუნველყოფის შესახებ, თუმცა ის მოითხოვს მეტ ადგილს ფლეშ მეხსიერება მეხსიერებაში. ის ასევე ინახავს გაფუჭებული აპლიკაციის სურათს SHA256. ბირთვის დამპინგის ეს ფორმატი რეკომენდებულია ახალი პროგრამული უზრუნველყოფის დიზაინისთვის და საკმარისად მოქნილია შენახული ინფორმაციის გასაფართოებლად მომავალი ვერსიებისთვის. ორობითი ფორმატი შენარჩუნებულია თავსებადობის თვალსაზრისით, ის ნაკლებ ადგილს იკავებს მეხსიერებაში მონაცემების შესანახად და უზრუნველყოფს უკეთეს მუშაობას.

  1. ძირითადი მონაცემების დამატების მაქსიმალური რაოდენობა (Components -> Core dump -> Maximum number of tasks).
  2. დაყოვნება ბირთვის ამონაწერის UART-ზე დაბეჭდვამდე (Components -> Core dump -> Delay before print to UART). მნიშვნელობა მილიწამშია.
  3. მონაცემთა მთლიანობის შემოწმების ტიპი ბირთვის დასამუხტისთვის (Components -> Core dump -> Core dump data integrity check).
  • გამოიყენეთ CRC32 ბირთვის დემპინგის მთლიანობის დასადასტურებლად
  • გამოიყენეთ SHA256 ბირთვის დემპინგის მთლიანობის დადასტურებისთვის

SHA256 ჰეშ ალგორითმი დაზიანების აღმოჩენის უფრო მეტ ალბათობას იძლევა, ვიდრე CRC32 მრავალბიტიანი შეცდომებით. CRC32 ვარიანტი უზრუნველყოფს უკეთეს გამოთვლების შესრულებას და ნაკლებ მეხსიერებას მოიხმარს შესანახად.

შეინახეთ ძირითადი დემპინგი ფლეშ მეხსიერება ში¶

როდესაც ეს პარამეტრი არჩეულია, core dump-ები ინახება ფლეშ მეხსიერება ის სპეციალურ დანაყოფში. როდესაც გამოიყენება ნაგულისხმევი დანაყოფების ცხრილი ფაილები, რომლებიც მოწოდებულია ESP-IDF ით, ის ავტომატურად გამოყოფს საჭირო ადგილს ფლეშ მეხსიერება ზე, მაგრამ თუ მომხმარებელს სურს გამოიყენოს საკუთარი განლაგების ფაილი core dump ფუნქციასთან ერთად, მან უნდა განსაზღვროს ცალკე დანაყოფი core dump-ისთვის, როგორც ეს ნაჩვენებია ქვემოთ:

# Name,   Type, SubType, Offset,  Size
# Note: if you have increased the bootloader size, make sure to update the offsets to avoid overlap
nvs,      data, nvs,     0x9000,  0x6000
phy_init, data, phy,     0xf000,  0x1000
factory,  app,  factory, 0x10000, 1M
coredump, data, coredump,,        64K

დანაყოფის სახელწოდებისთვის განსაკუთრებული მოთხოვნები არ არსებობს. მისი არჩევა შესაძლებელია მომხმარებლის აპლიკაციის საჭიროებების შესაბამისად, მაგრამ დანაყოფის ტიპი უნდა იყოს „მონაცემები“ და ქვეტიპი უნდა იყოს „coredump“. ასევე, დანაყოფის ზომის არჩევისას გაითვალისწინეთ, რომ core dump მონაცემთა სტრუქტურა ითვალისწინებს 20 ბაიტის მუდმივ დატვირთვას და 12 ბაიტის დატვირთვას თითოეული დავალებისთვის. ეს დატვირთვა არ მოიცავს თითოეული დავალებისთვის TCB-ის და სტეკის ზომას. ამიტომ, დანაყოფის ზომა უნდა იყოს მინიმუმ 20 + მაქსიმალური დავალებების რაოდენობა x (12 + TCB ზომა + მაქსიმალური დავალებების სტეკის ზომა) ბაიტი.

ფლეშ მეხსიერება დან core dump-ის ანალიზის ზოგადი ბრძანების მაგალითია: espcoredump.py -p </path/to/serial/port> info_corefile </path/to/program/elf/file> ან espcoredump.py -p </path/to/serial/port> dbg_corefile </path/to/program/elf/file>

ბირთვის ამონაწერის UART-ზე დაბეჭდვა¶

როდესაც ეს პარამეტრი არჩეულია base64-ით კოდირებული ბირთვის დემპები იბეჭდება UART-ზე სისტემის პანიკის შემთხვევაში. ამ შემთხვევაში, მომხმარებელმა ხელით უნდა შეინახოს ბირთვის დემპის ტექსტი რომელიმე ფაილში და შემდეგ გაუშვას შემდეგი ბრძანება: espcoredump.py info_corefile -t b64 -c </path/to/saved/base64/text> </path/to/program/elf/file> ან espcoredump.py dbg_corefile -t b64 -c </path/to/saved/base64/text> </path/to/program/elf/file>

Base64-კოდირებული ბირთვის დემპინგის ძირითადი ფაილი მოთავსდება შემდეგ სათაურსა და ქვედა კოლონტიტულს შორის:

================= CORE DUMP START =================
<body of base64-encoded core dump, save it to file on disk>
================= CORE DUMP END ===================

ის CORE DUMP START და CORE DUMP END ხაზები არ უნდა იყოს შეტანილი core dump ტექსტურ ფაილში.

ROM ფუნქციები Backtraces-ში¶

შესაძლებელია, რომ ავარიის მომენტში ზოგიერთ დავალებას ან/და თავად ავარიულ დავალებას ჰქონდეს ერთი ან მეტი ROM ფუნქცია მათ გამოძახების სტეკებში. რადგან ROM არ არის პროგრამის ELF ნაწილი, GDB-სთვის შეუძლებელი იქნება ასეთი გამოძახების სტეკების გაანალიზება, რადგან ის ცდილობს ფუნქციების პროლოგების ანალიზს ამის მისაღწევად. ამ შემთხვევაში გამოძახების სტეკის ბეჭდვა შეწყდება შეცდომის შეტყობინებით პირველ ROM ფუნქციაზე. ამ პრობლემის გადასაჭრელად შეგიძლიათ გამოიყენოთ Espressif-ის მიერ მოწოდებული ROM ELF (https://dl.espressif.com/dl/esp32_rom.elf) და გადაეცით ის &#39;espcoredump.py&#39;-ს.

&#39;espcoredump.py&#39;-ის გაშვება¶

ზოგადი ბრძანების სინტაქსი:

espcoredump.py [options] command [args]

სკრიპტის პარამეტრები:
  • –chip,-c {auto,esp32}. სამიზნე ჩიპის ტიპი. მხარდაჭერილი მნიშვნელობებია auto და esp32.
  • –port,-p PORT. სერიული პორტის მოწყობილობა.
  • –baud, -b BAUD. სერიული პორტის ბოდის სიჩქარე გამოიყენება ფლეშ მეხსიერება წაკითხვისას/ჩაწერისას.
ბრძანებები:
  • info_corefile. ბირთვის ფაილის ამოღება და სასარგებლო ინფორმაციის დაბეჭდვა.
  • dbg_corefile. core dump-ის მოძიება და GDB სესიის დაწყება.
ბრძანების არგუმენტები:
 
  • – გამართვა , -d DEBUG. ჟურნალის დონე (0..3).
  • –gdb,-g GDB. gdb-ის გზა, რომელიც გამოიყენება მონაცემების მოსაძიებლად.
  • –core,-c CORE. გამოსაყენებელი core dump ფაილის გზა (გამოტოვების შემთხვევაში core dump წაიკითხება ფლეშ მეხსიერება დან).
  • –core-format, -t CORE_FORMAT. მიუთითებს, რომ „-c“-თი გადაცემული ფაილი არის ELF („elf“), დამპინგებული ნედლი ორობითი („raw“) ან base64-კოდირებული („b64“) ფორმატით.
  • –o,-o OFF. coredump დანაყოფის ოფსეტი ფლეშ მეხსიერება ში (ტიპი idf.py partition_table რომ ნახოთ).
  • –save-core,-s SAVE_CORE. შეინახეთ ბირთვი ფაილში. წინააღმდეგ შემთხვევაში, დროებითი ბირთვის ფაილი წაიშლება. იგნორირებულია „-c“-ით.
  • –rom-elf,-r ROM_ELF. გამოსაყენებელი ROM ELF ფაილის გზა (გამოტოვების შემთხვევაში გამოყენებული იქნება „esp32_rom.elf“).
  • –print-mem, -m მეხსიერების დემპინგის ბეჭდვა. გამოიყენება მხოლოდ „info_corefile“-თან ერთად.
  • <prog>პროგრამის ELF ფაილისკენ მიმავალი გზა.

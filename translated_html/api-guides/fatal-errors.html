ფატალური შეცდომები¶

[中文]

მიმოხილვა¶

გარკვეულ სიტუაციებში, პროგრამის შესრულების გაგრძელება კარგად განსაზღვრული გზით შეუძლებელია. ESP-IDF -ში ეს სიტუაციები მოიცავს:

  • CPU-ს გამონაკლისები: უკანონო ინსტრუქცია, ჩატვირთვის/შენახვის გასწორების შეცდომა, ჩატვირთვის/შენახვის აკრძალული შეცდომა, ორმაგი გამონაკლისი.
  • სისტემური დონის შემოწმებები და დაცვის ზომები:
    • შეწყვეტის მცველი ტაიმ-აუტი
    • დავალების მაკონტროლებელი ტაიმ-აუტი (მხოლოდ ფატალურია, თუ CONFIG_ESP_TASK_WDT_PANIC დაყენებულია)
    • ქეშზე წვდომის შეცდომა
    • ძაბვის ვარდნის აღმოჩენის მოვლენა
    • დასტის გადავსება
    • დასტის დარტყმისგან დაცვის შემოწმება
    • გროვის მთლიანობის შემოწმება
  • წარუმატებელი მტკიცებები, მეშვეობით assert, configASSERT და მსგავსი მაკროები.

ეს სახელმძღვანელო განმარტავს ESP-IDF ში ამ შეცდომების მოსაგვარებლად გამოყენებულ პროცედურას და იძლევა რჩევებს შეცდომების გამოსწორების შესახებ.

პანიკის მართვის სპეციალისტი¶

ყველა შეცდომის მიზეზი, რომელიც ჩამოთვლილია მიმოხილვა დამუშავდება პანიკის მართვა.

პანიკის დამმუშავებელი დაიწყება შეცდომის მიზეზის კონსოლზე დაბეჭდვით. CPU-ს გამონაკლისების შემთხვევაში, შეტყობინება მსგავსი იქნება:

Guru Meditation Error: Core 0 panic'ed (IllegalInstruction). Exception was unhandled.

სისტემის დონის ზოგიერთი შემოწმების შემთხვევაში (შეფერხების მაკონტროლებელი, ქეშზე წვდომის შეცდომა), შეტყობინება მსგავსი იქნება:

Guru Meditation Error: Core 0 panic'ed (Cache disabled but cached memory region accessed)

ყველა შემთხვევაში, შეცდომის მიზეზი ფრჩხილებში იქნება დაბეჭდილი. იხილეთ გურუს მედიტაციის შეცდომები შესაძლო შეცდომების მიზეზების სიისთვის.

პანიკის დამმუშავებლის შემდგომი ქცევის დაყენება შესაძლებელია გამოყენებით CONFIG_ESP32_PANIC კონფიგურაციის არჩევანი. ხელმისაწვდომი ვარიანტებია:

  • რეგისტრების დაბეჭდვა და გადატვირთვა (CONFIG_ESP32_PANIC_PRINT_REBOOT) — ნაგულისხმევი ვარიანტი.

    ეს დაბეჭდავს რეგისტრის მნიშვნელობებს გამონაკლისის წერტილში, დაბეჭდავს უკუკავშირს და გადატვირთავს ჩიპს.

  • რეგისტრების ბეჭდვა და შეჩერება (CONFIG_ESP32_PANIC_PRINT_HALT)

    ზემოთ მოცემული ვარიანტის მსგავსი, მაგრამ გადატვირთვის ნაცვლად შეჩერება. პროგრამის გადასატვირთად საჭიროა გარე გადატვირთვა.

  • ჩუმი გადატვირთვა (CONFIG_ESP32_PANIC_SILENT_REBOOT)

    არ დაბეჭდოთ რეგისტრები და არ გამოიყენოთ უკუკავშირი, დაუყოვნებლივ გადატვირთეთ ჩიპი.

  • GDB Stub-ის გამოძახება (CONFIG_ESP32_PANIC_GDBSTUB)

    გაუშვით GDB სერვერი, რომელსაც შეუძლია GDB-სთან კომუნიკაცია კონსოლის UART პორტის საშუალებით. იხილეთ GDB Stub დამატებითი დეტალებისთვის.

პანიკის დამმუშავებლის ქცევაზე გავლენას ახდენს კონფიგურაციის ორი სხვა ვარიანტი.

  • თუ CONFIG_ESP32_DEBUG_OCDAWARE თუ ჩართულია (რაც ნაგულისხმევია), პანიკის დამმუშავებელი აღმოაჩენს, დაკავშირებულია თუ არა JTAG გამართვა ger. თუ ასეა, შესრულება შეჩერდება და კონტროლი გადაეცემა გამართვა ger-ს. ამ შემთხვევაში რეგისტრები და უკუკავშირი არ იგზავნება კონსოლზე და GDBStub / Core Dump ფუნქციები არ გამოიყენება.

  • თუ ბირთვის ნაგავსაყრელი ფუნქცია ჩართულია (CONFIG_ESP32_ENABLE_COREDUMP_TO_FLASH ან CONFIG_ESP32_ENABLE_COREDUMP_TO_UART პარამეტრები), შემდეგ სისტემის მდგომარეობა (დავალებების დასტები და რეგისტრები) გადაიცემა Flash-ზე ან UART-ზე, შემდგომი ანალიზისთვის.

  • თუ CONFIG_ESP_PANIC_HANDLER_IRAM თუ გამორთულია (ნაგულისხმევად გამორთულია), პანიკის დამმუშავებლის კოდი მოთავსებულია ფლეშ მეხსიერება მეხსიერებაში და არა IRAM-ში. ეს ნიშნავს, რომ თუ ESP-IDF გაითიშება, ხოლო ფლეშ მეხსიერება ქეში გამორთულია, პანიკის დამმუშავებელი ავტომატურად ხელახლა ჩართავს ფლეშ მეხსიერება ქეშს GDB Stub-ის ან Core Dump-ის გაშვებამდე. ეს ზრდის გარკვეულ მცირე რისკს, თუ ავარიის დროს ფლეშ მეხსიერება ქეშის სტატუსიც დაზიანდება.

    თუ ეს პარამეტრი ჩართულია, პანიკის დამმუშავებლის კოდი თავსდება IRAM-ში. ეს საშუალებას აძლევს პანიკის დამმუშავებელს იმუშაოს ქეშის ხელახლა ჩართვის გარეშე. ეს შეიძლება საჭირო იყოს გამართვა ავარიებთან დაკავშირებული ზოგიერთი რთული პრობლემის გადასაჭრელად, როდესაც ფლეშ მეხსიერება ქეში გამორთულია (მაგალითად, SPI ფლეშ მეხსიერება ში ჩაწერისას).

შემდეგი დიაგრამა ასახავს პანიკის მოპყრობის მქონე პირთა ქცევას:

პანიკის მართვის ბლოკ-სქემა (გადიდებისთვის დააწკაპუნეთ)¶

რეგისტრაციის დემპინგი და უკუკვლევა¶

დაბეჭდილი რეგისტრის მნიშვნელობები არის გამონაკლისის ჩარჩოში არსებული რეგისტრის მნიშვნელობები, ანუ მნიშვნელობები იმ მომენტში, როდესაც მოხდა CPU გამონაკლისი ან სხვა ფატალური შეცდომა.

რეგისტრის დემპინგი არ იბეჭდება, თუ პანიკის დამმუშავებელი შესრულდა შედეგად. abort() ზარი.

ზოგიერთ შემთხვევაში, მაგალითად, შეწყვეტის მაკონტროლებელი სისტემის დროის ამოწურვისას, პანიკის დამმუშავებელმა შეიძლება დაბეჭდოს დამატებითი CPU რეგისტრები (EPC1-EPC4) და სხვა CPU-ზე გაშვებული კოდის რეგისტრები/უკუკვლევა.

უკუკვლევადობის ხაზი შეიცავს PC:SP წყვილებს, სადაც PC არის პროგრამის მთვლელი და SP არის სტეკის მაჩვენებელი, მიმდინარე დავალების თითოეული სტეკის ჩარჩოსთვის. თუ ISR-ში ფატალური შეცდომა მოხდება, უკუკვლევადობა შეიძლება მოიცავდეს PC:SP წყვილებს როგორც შეწყვეტილი დავალებიდან, ასევე ISR-დან.

თუ IDF მონიტორი გამოყენების შემთხვევაში, პროგრამის მრიცხველის მნიშვნელობები გარდაიქმნება კოდის ადგილმდებარეობებად (ფუნქციის სახელი, ფაილის სახელი და ხაზის ნომერი) და გამომავალს დაემატება დამატებითი ხაზები:

შენიშვნა

იხილეთ ვიკიპედიის სტატია „გურუ მედიტაციის“ ისტორიული წარმოშობისთვის.

უკანონო ინსტრუქცია¶

ეს CPU გამონაკლისი მიუთითებს, რომ შესრულებული ინსტრუქცია არ იყო სწორი ინსტრუქცია. ამ შეცდომის ყველაზე გავრცელებული მიზეზებია:

  • Free RTOS დავალების ფუნქცია დაბრუნდა. Free RTOS ში, თუ დავალების ფუნქციას სჭირდება შეწყვეტა, მან უნდა გამოიძახოს vTaskDelete() ფუნქცია და წაშალოს საკუთარი თავი, დაბრუნების ნაცვლად.
  • SPI ფლეშ მეხსიერება დან შემდეგი ინსტრუქციის ჩატვირთვა ვერ მოხერხდა. ეს ჩვეულებრივ ხდება, თუ:
    • აპლიკაციამ SPI ფლეშ მეხსიერება პინები სხვა ფუნქციის სახით ( GPIO , UART და ა.შ.) ხელახლა დააკონფიგურირა. SPI ფლეშ მეხსიერება პინების შესახებ დამატებითი ინფორმაციისთვის იხილეთ ჩიპის ან მოდულის აპარატურის დიზაინის სახელმძღვანელო პრინციპები და მონაცემთა ცხრილი.
    • ზოგიერთი გარე მოწყობილობა შემთხვევით დაუკავშირდა SPI ფლეშ მეხსიერება პინებს და ხელი შეუშალა ESP32-სა და SPI ფლეშ მეხსიერება შორის კომუნიკაციას.

InstrFetchაკრძალულია¶

ეს CPU გამონაკლისი მიუთითებს, რომ CPU-ს არ შეეძლო ინსტრუქციის ჩატვირთვა, რადგან ინსტრუქციის მისამართი არ ეკუთვნოდა ინსტრუქციის RAM-ის ან ROM-ის ვალიდურ რეგიონს.

როგორც წესი, ეს ნიშნავს ფუნქციის მაჩვენებლის გამოძახების მცდელობას, რომელიც ვალიდურ კოდზე არ მიუთითებს. PC (პროგრამის მრიცხველის) რეგისტრი შეიძლება გამოყენებულ იქნას როგორც ინდიკატორი: ის იქნება ნული ან შეიცავს ნაგვის მნიშვნელობას (არა 0x4xxxxxxx).

ჩატვირთვა აკრძალულია, შენახვა აკრძალულია¶

ეს CPU გამონაკლისი ხდება მაშინ, როდესაც აპლიკაცია ცდილობს არასწორი მეხსიერების მდებარეობიდან წაკითხვას ან ჩაწერას. ჩაწერილი/წაკითხული მისამართი მოიძებნება შემდეგში: EXCVADDR რეგისტრაცია რეგისტრის დემპში. თუ ეს მისამართი ნულის ტოლია, ეს ჩვეულებრივ ნიშნავს, რომ აპლიკაციამ სცადა NULL მაჩვენებლის დერეფერენსირება. თუ ეს მისამართი ნულთან ახლოსაა, ეს ჩვეულებრივ ნიშნავს, რომ აპლიკაციამ სცადა სტრუქტურის წევრზე წვდომა, მაგრამ სტრუქტურის მაჩვენებელი NULL იყო. თუ ეს მისამართი სხვა რამ არის (ნაგვის მნიშვნელობა, არა 0x3fxxxxxx - 0x6xxxxxxx დიაპაზონი), ეს სავარაუდოდ ნიშნავს, რომ მონაცემებზე წვდომისთვის გამოყენებული მაჩვენებელი ან არ იყო ინიციალიზებული, ან დაზიანებული იყო.

მთელი რიცხვის გაყოფა ნულზე¶

აპლიკაციამ სცადა მთელი რიცხვის ნულზე გაყოფა.

LoadStoreAlignment¶

აპლიკაციამ სცადა მეხსიერების მდებარეობის წაკითხვა ან ჩაწერა და მისამართის გასწორება არ ემთხვეოდა ჩატვირთვას/შენახვის ზომას. მაგალითად, 32-ბიტიანი ჩატვირთვა შესაძლებელია მხოლოდ 4-ბაიტიანი გასწორებული მისამართიდან, ხოლო 16-ბიტიანი ჩატვირთვა შესაძლებელია მხოლოდ 2-ბაიტიანი გასწორებული მისამართიდან.

აპლიკაციამ სცადა 8 ან 16 ბიტიანი ჩატვირთვა/შენახვის განხორციელება მეხსიერების რეგიონიდან, რომელიც მხოლოდ 32 ბიტიან ჩატვირთვას/შენახვას უჭერს მხარს. მაგალითად, დერეფერენსინგი char* მაჩვენებელი, რომელიც მიუთითებს ინსტრუქციის მეხსიერებაზე, გამოიწვევს ასეთ შეცდომას.

დაუმუშავებელი გამართვა გამონაკლისი¶

ამას, როგორც წესი, მოჰყვება შეტყობინება, როგორიცაა:

CPU0 / CPU1-ზე wdt-ის შეწყვეტის დროის ამოწურვა¶

მიუთითებს, რომ შეწყვეტის მაკონტროლებელი დროის ამოწურვა მოხდა. იხილეთ მეთვალყურე ძაღლები დამატებითი ინფორმაციისთვის.

ქეში გამორთულია, მაგრამ ქეშირებული მეხსიერების რეგიონი განხილულია¶

ზოგიერთ სიტუაციაში ESP-IDF დროებით გამორთავს წვდომას გარე SPI Flash-სა და SPI RAM-ზე ქეშის საშუალებით. მაგალითად, ეს ხდება spi_ ფლეშ მეხსიერება შემთხვევაში. API გამოიყენება SPI Flash-ის რეგიონების წასაკითხად/ჩასაწერად/წასაშლელად/mmap-ის შესაქმნელად. ასეთ სიტუაციებში, დავალებები შეჩერებულია და შეფერხების დამმუშავებლები არ არის რეგისტრირებული. ESP_INTR_FLAG_IRAM გამორთულია. დარწმუნდით, რომ ამ დროშით რეგისტრირებულ ნებისმიერ შეფერხების დამმუშავებელს აქვს IRAM/DRAM-ში არსებული ყველა კოდი და მონაცემი. იხილეთ SPI-ის ფლეშ მეხსიერება API დოკუმენტაცია დამატებითი დეტალებისთვის.

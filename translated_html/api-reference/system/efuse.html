eFuse მენეჯერი¶

შესავალი¶

eFuse Manager-ის ბიბლიოთეკა შექმნილია eFuse ბიტებზე წვდომის სტრუქტურირებისა და მათი გამოყენების გამარტივებისთვის. ეს ბიბლიოთეკა eFuse ბიტებს მართავს სტრუქტურის სახელით, რომელიც მინიჭებულია eFuse ცხრილში. ეს ნაწილი წარმოგიდგენთ eFuse Manager-ის მიერ გამოყენებულ ზოგიერთ კონცეფციას.

აპარატურის აღწერა¶

ESP32-ს აქვს რამდენიმე eFuse, რომლებსაც შეუძლიათ სისტემის და მომხმარებლის პარამეტრების შენახვა. თითოეული eFuse არის ერთბიტიანი ველი, რომლის დაპროგრამება შესაძლებელია 1-ზე, რის შემდეგაც მისი 0-ზე დაბრუნება შეუძლებელია. სისტემის ზოგიერთი პარამეტრი იყენებს ამ eFuse ბიტებს პირდაპირ აპარატურის მოდულების მიერ და მათ განსაკუთრებული ადგილი უკავია (მაგალითად, EFUSE_BLK0). დამატებითი ინფორმაციისთვის იხილეთ ESP32 ტექნიკური საცნობარო სახელმძღვანელო მე-20 ნაწილში eFuse კონტროლერი. ზოგიერთი eFuse ბიტი ხელმისაწვდომია მომხმარებლის აპლიკაციებისთვის.

ESP32-ს აქვს 4 eFuse ბლოკი, რომელთაგან თითოეული 256 ბიტის ზომისაა (ყველა ბიტი არ არის ხელმისაწვდომი):

  • EFUSE_BLK0 გამოიყენება მთლიანად სისტემის მიზნებისთვის;
  • EFUSE_BLK1 გამოიყენება ფლეშ მეხსიერება დაშიფვრის გასაღებისთვის. თუ არ იყენებთ ამ Flash დაშიფვრის ფუნქციას, მათი გამოყენება სხვა მიზნით შეიძლება;
  • EFUSE_BLK2 გამოიყენება უსაფრთხოების ჩატვირთვის გასაღებისთვის. თუ არ იყენებთ ამ უსაფრთხო ჩატვირთვის ფუნქციას, მათი გამოყენება სხვა მიზნით შეიძლება;
  • EFUSE_BLK3 შეიძლება ნაწილობრივ იყოს დაჯავშნილი მორგებული MAC მისამართისთვის, ან მთლიანად გამოყენებული იყოს მომხმარებლის აპლიკაციისთვის. გაითვალისწინეთ, რომ ზოგიერთი ბიტი უკვე გამოიყენება IDF-ში.

თითოეული ბლოკი დაყოფილია 8 32-ბიტიან რეგისტრად.

eFuse მენეჯერი კომპონენტი¶

კომპონენტი აქვს API ფუნქციები ველების წასაკითხად და ჩასაწერად. ველებზე წვდომა ხორციელდება სტრუქტურების მეშვეობით, რომლებიც აღწერენ eFuse ბიტების მდებარეობას ბლოკებში. კომპონენტი იძლევა ნებისმიერი სიგრძისა და ინდივიდუალური ბიტების ნებისმიერი რაოდენობის ველების ფორმირების შესაძლებლობას. ველების აღწერა კეთდება CSV ფაილში ცხრილის სახით. C-წყაროში ცხრილის ფორმიდან (CSV ფაილი) გენერირებისთვის გამოიყენება ინსტრუმენტი efuse_table_gen.pyინსტრუმენტი ამოწმებს CSV ფაილს ველის სახელების უნიკალურობასა და ბიტების გადაკვეთაზე, იმ შემთხვევაში, თუ გამოიყენება custom ფაილი მომხმარებლის პროექტის დირექტორიიდან, პროგრამა შეამოწმებს common CSV ფაილი.

CSV ფაილები:

  • საერთო (esp_efuse_table.csv) - შეიცავს eFuse ველებს, რომლებიც გამოიყენება IDF-ის შიგნით. C-წყაროს გენერირება უნდა განხორციელდეს ხელით ამ ფაილის შეცვლისას (run ბრძანება idf.py efuse_common_table). გაითვალისწინეთ, რომ ამ ფაილში ცვლილებებმა შეიძლება არასწორი ოპერაცია გამოიწვიოს.
  • მორგებული - (არასავალდებულო და შეიძლება ჩართოთ CONFIG_EFUSE_CUSTOM_TABLE) შეიცავს eFuse ველებს, რომლებსაც მომხმარებელი იყენებს მის აპლიკაციაში. C-წყაროს გენერირება უნდა განხორციელდეს ხელით ამ ფაილის შეცვლისა და გაშვებისას. idf.py efuse_custom_table.

აღწერა CSV ფაილი¶

CSV ფაილი შეიცავს eFuse ველების აღწერას. მარტივ შემთხვევაში, ერთ ველს აქვს ერთი ხაზი აღწერისთვის. ცხრილის სათაური:

# field_name,  efuse_block(EFUSE_BLK0..EFUSE_BLK3), bit_start(0..255),  bit_count(1..256),              comment

CSV ფაილში ცალკეულ პარამეტრებს შემდეგი მნიშვნელობა აქვთ:

field_name
Name of field. The prefix ESP_EFUSE_ will be added to the name, and this field name will be available in the code. This name will be used to access the fields. The name must be unique for all fields. If the line has an empty name, then this line is combined with the previous field. This allows you to set an arbitrary order of bits in the field, and expand the field as well (see MAC_FACTORY field in the common table).
efuse_block
Block number. It determines where the eFuse bits will be placed for this field. Available EFUSE_BLK0..EFUSE_BLK3.
bit_start
Start bit number (0..255). The bit_start field can be omitted. In this case, it will be set to bit_start + bit_count from the previous record, if it has the same efuse_block. Otherwise (if efuse_block is different, or this is the first entry), an error will be generated.
bit_count
The number of bits to use in this field (1..-). This parameter can not be omitted. This field also may be MAX_BLK_LEN in this case, the field length will have the maximum block length, taking into account the coding scheme (applicable for ESP_EFUSE_SECURE_BOOT_KEY and ESP_EFUSE_ENCRYPT_FLASH_KEY fields). The value MAX_BLK_LEN depends on CONFIG_EFUSE_MAX_BLK_LEN, will be replaced with “None” - 256, “3/4” - 192, “REPEAT” - 128.
comment
This param is using for comment field, it also move to C-header file. The comment field can be omitted.

თუ ველის აღსაწერად საჭიროა არათანმიმდევრული ბიტების თანმიმდევრობა, მაშინ ველის აღწერა შემდეგ სტრიქონებში უნდა გაგრძელდეს სახელის მითითების გარეშე, რაც მიუთითებს, რომ ის ერთ ველს ეკუთვნის. მაგალითად, ორი ველი MAC_FACTORY და MAC_FACTORY_CRC:

# Factory MAC address #
#######################
MAC_FACTORY,            EFUSE_BLK0,    72,    8,    Factory MAC addr [0]
,                       EFUSE_BLK0,    64,    8,    Factory MAC addr [1]
,                       EFUSE_BLK0,    56,    8,    Factory MAC addr [2]
,                       EFUSE_BLK0,    48,    8,    Factory MAC addr [3]
,                       EFUSE_BLK0,    40,    8,    Factory MAC addr [4]
,                       EFUSE_BLK0,    32,    8,    Factory MAC addr [5]
MAC_FACTORY_CRC,        EFUSE_BLK0,    80,    8,    CRC8 for factory MAC address

ეს ველი კოდში ხელმისაწვდომი იქნება როგორც ESP_EFUSE_MAC_FACTORY და ESP_EFUSE_MAC_FACTORY_CRC.

efuse_table_gen.py ინსტრუმენტი¶

ინსტრუმენტი შექმნილია CSV ფაილიდან C-წყაროს ფაილების გენერირებისა და ველების დასადასტურებლად. უპირველეს ყოვლისა, მოწმდება სახელების უნიკალურობა და ველის ბიტების გადაფარვა. თუ დამატებითი custom ფაილი გამოიყენება, ის შემოწმდება არსებულთან common ფაილი (esp_efuse_table.csv). შეცდომების შემთხვევაში, გამოჩნდება შეტყობინება და შეცდომის გამომწვევი სტრიქონი. C-წყაროს ფაილები შეიცავს ტიპის სტრუქტურებს. esp_efuse_desc_t.

გენერირებისთვის common ფაილები, გამოიყენეთ შემდეგი ბრძანება idf.py efuse_common_table ან:

cd $IDF_PATH/components/efuse/
./efuse_table_gen.py esp32/esp_efuse_table.csv

საქაღალდეში გენერირების შემდეგ esp32 შექმნა:

  • esp_efuse_table.c ფაილი.
  • შიგნით include საქაღალდე esp_efuse_table.c ფაილი.

გენერირებისთვის custom ფაილები, გამოიყენეთ შემდეგი ბრძანება idf.py efuse_custom_table ან:

cd $IDF_PATH/components/efuse/
./efuse_table_gen.py esp32/esp_efuse_table.csv PROJECT_PATH/main/esp_efuse_custom_table.csv

PROJECT_PATH/main საქაღალდეში შექმნის შემდეგ:

  • esp_efuse_custom_table.c ფაილი.
  • შიგნით include საქაღალდე esp_efuse_custom_table.c ფაილი.

გენერირებული ველების გამოსაყენებლად, თქვენ უნდა დაამატოთ ორი ფაილი:

#include "esp_efuse.h"
#include "esp_efuse_table.h" or "esp_efuse_custom_table.h"

კოდირების სქემის მხარდაჭერა¶

eFuse-ს აქვს სამი კოდირების სქემა:

  • None (მნიშვნელობა 0).
  • 3/4 (მნიშვნელობა 1).
  • Repeat (მნიშვნელობა 2).

კოდირების სქემა მოქმედებს მხოლოდ EFUSE_BLK1, EFUSE_BLK2 და EFUSE_BLK3 ბლოკებზე. EUSE_BLK0 ბლოკს ყოველთვის აქვს კოდირების სქემა. Noneკოდირება ცვლის ბლოკში ჩასაწერი ბიტების რაოდენობას, ბლოკის სიგრძე მუდმივია 256, ამ ბიტების ნაწილი გამოიყენება კოდირებისთვის და არ გამოიყენება.

კოდირების სქემის გამოყენებისას, ჩასაწერი დატვირთვის სიგრძე შეზღუდულია (დამატებითი ინფორმაციისთვის 20.3.1.3 System Parameter coding_scheme):

  • არცერთი 256 ბიტიანი.
  • 3/4 192 ბიტი.
  • გაიმეორეთ 128 ბიტი.

შეგიძლიათ გაიგოთ თქვენი ჩიპის კოდირების სქემა:

  • გაუშვით espefuse.py -p COM4 summary ბრძანება.
  • -დან esptool კომუნალური მომსახურების ჟურნალები ( ფლეშ მეხსიერება სემესტრის განმავლობაში).
  • კოდში ფუნქციის გამოძახება esp_efuse_get_coding_scheme() EFUSE_BLK3 ბლოკისთვის.

eFuse ცხრილები ყოველთვის უნდა შეესაბამებოდეს ჩიპში არსებულ კოდირების სქემას. არსებობს EFUSE_CODE_SCHEME_SELECTOR Kconfig-ში ცხრილების კოდირების ტიპის შესარჩევად ვარიანტი. საწყისი ფაილების გენერირებისას, თუ თქვენი ცხრილები არ მიჰყვება კოდირების სქემას, გამოჩნდება შეცდომის შეტყობინება. შეცვალეთ სიგრძის ან ოფსეტის ველები. თუ თქვენი პროგრამა იყო კომპილაცია d-ით None კოდირება და 3/4 გამოიყენება ჩიპში, შემდეგ კი ESP_ERR_CODING eFuse API გამოძახებისას შეიძლება შეცდომა წარმოიშვას (ველი ბლოკის საზღვრებს გარეთაა). თუ ველი ემთხვევა ახალი ბლოკის საზღვრებს, მაშინ API შეცდომების გარეშე იმუშავებს.

ასევე, 3/4 კოდირების სქემა აწესებს შეზღუდვებს ერთი კოდირების ერთეულის კუთვნილი ბიტების ჩაწერაზე. 256 ბიტის სიგრძის მთელი ბლოკი დაყოფილია 4 კოდირების ერთეულად და თითოეულ კოდირების ერთეულში არის 6 ბაიტი სასარგებლო მონაცემები და 2 სერვისული ბაიტი. ეს 2 სერვისული ბაიტი შეიცავს წინა 6 მონაცემთა ბაიტის ჩეკსუმს.

გამოდის, რომ ერთ კოდირების ერთეულში მხოლოდ ერთი ველის ჩაწერაა შესაძლებელი. ერთ კოდირების ერთეულში განმეორებითი გადაწერა აკრძალულია. თუმცა, თუ ჩანაწერი წინასწარ ან სხვა საშუალებით გაკეთდა. esp_efuse_write_block() ფუნქცია, მაშინ შესაძლებელია ერთი კოდირების ერთეულის კუთვნილი ველების წაკითხვა.

იმ შემთხვევაში, თუ 3/4 კოდირების სქემის მიხედვით, ჩაწერის პროცესი დაყოფილია კოდირების ერთეულებად და ჩვენ არ შეგვიძლია გამოვიყენოთ ზოგიერთი ველის ჩაწერის ჩვეულებრივი რეჟიმი. ჩვენ შეგვიძლია მოვამზადოთ ყველა მონაცემი ჩასაწერად და ერთდროულად ჩავწეროთ. ასევე შეგიძლიათ გამოიყენოთ ეს რეჟიმი None კოდირების სქემა, მაგრამ ეს აუცილებელი არ არის. მნიშვნელოვანია 3/4 კოდირების სქემა. რამდენიმე ველის ერთდროულად დასაწერად საჭიროა გამოვიყენოთ the batch writing modeპირველ რიგში, დააყენეთ ეს რეჟიმი esp_efuse_batch_write_begin() ფუნქცია, შემდეგ ჩაწერეთ რამდენიმე ველი ჩვეულებისამებრ გამოიყენეთ esp_efuse_write_... ფუნქციები. მათი ჩაწერის ბოლოს, საჭიროა გამოიძახოთ esp_efuse_batch_write_commit() ფუნქცია. ის წვავს მომზადებულ მონაცემებს გაყალბების ბლოკებში და გამორთავს batch recording mode. The batch writing mode ბლოკები esp_efuse_read_... ოპერაციები.

კოდირების სქემის შეცვლის შემდეგ, გაუშვით efuse_common_table და efuse_custom_table ბრძანებები ახალი კოდირების სქემის ცხრილების შესამოწმებლად.

eFuse API¶

ველებზე წვდომა აღწერილობის სტრუქტურაზე მითითების საშუალებით ხდება. API ფუნქციებს აქვთ რამდენიმე ძირითადი ოპერაცია:

  • esp_efuse_read_field_blob() - აბრუნებს წაკითხული eFuse ბიტების მასივს.
  • esp_efuse_read_field_cnt() - აბრუნებს დაპროგრამებული ბიტების რაოდენობას, როგორც „1“.
  • esp_efuse_write_field_blob() - წერს მასივს.
  • esp_efuse_write_field_cnt() - საჭირო რაოდენობის ბიტებს წერს, როგორც „1“.
  • esp_efuse_get_field_size() - აბრუნებს ველის სახელის მიხედვით ბიტების რაოდენობას.
  • esp_efuse_read_reg() - აბრუნებს eFuse რეგისტრის მნიშვნელობას.
  • esp_efuse_write_reg() - წერს მნიშვნელობას eFuse რეგისტრში.
  • esp_efuse_get_coding_scheme() - ბლოკებისთვის აბრუნებს eFuse კოდირების სქემას.
  • esp_efuse_read_block() - კითხულობს eFuse ბლოკის გასაღებს, დაწყებული ოფსეტიდან და საჭირო ზომიდან.
  • esp_efuse_write_block() - eFuse ბლოკში გასაღების ჩაწერა, ოფსეტიდან და საჭირო ზომიდან დაწყებული.
  • esp_efuse_batch_write_begin() - დააყენეთ ველების ჩაწერის პაკეტური რეჟიმი.
  • esp_efuse_batch_write_commit() - იწერს ყველა მომზადებულ მონაცემს პაკეტური ჩაწერის რეჟიმისთვის და აღადგენს პაკეტური ჩაწერის რეჟიმს.
  • esp_efuse_batch_write_cancel() - გადატვირთეთ პაკეტური ჩაწერის რეჟიმი და მომზადებული მონაცემები.

ხშირად გამოყენებული ველებისთვის, სპეციალური ფუნქციები იქმნება, მაგალითად, ეს esp_efuse_get_chip_ver(), esp_efuse_get_pkg_ver().

ვირტუალური eFuses¶

Kconfig-ის ვარიანტი CONFIG_EFUSE_VIRTUAL eFuse Manager-ში eFuse მნიშვნელობების ვირტუალიზაცია მოხდება, რათა ჩაწერები ემულირებული იყოს და eFuse მნიშვნელობები სამუდამოდ არ შეიცვალოს. ეს შეიძლება სასარგებლო იყოს გამართვა ging აპლიკაციისა და ერთეულის ტესტებისთვის.

ფუნქციები¶

esp_err_t esp_efuse_read_field_blob(კონსტ esp_efuse_desc_t *ველი[], void *თარიღი, size_t dst_size_bits)¶

კითხულობს ბიტებს EFUSE ველიდან და წერს მათ მასივში.

წაკითხული ბიტების რაოდენობა შემოიფარგლება „field“ სტრუქტურაში ბიტების აღწერილობიდან მინიმალური მნიშვნელობით ან „dst_size_bits“ საჭირო ზომით. ველის სიგრძის დასადგენად გამოიყენეთ „esp_efuse_get_field_size()“ ფუნქცია.

დაბრუნება
  • ESP_OK: ოპერაცია წარმატებით დასრულდა.
  • ESP_ERR_INVALID_ARG: შეცდომა გადაცემულ არგუმენტებში.
პარამეტრები
  • fieldefuse-ის ველების აღმწერი სტრუქტურის მაჩვენებელი.
  • dst: მასივის მაჩვენებელი, რომელიც შეიცავს წაკითხვის შედეგს.
  • dst_size_bitsწასაკითხად საჭირო ბიტების რაოდენობა. თუ მოთხოვნილი ბიტების რაოდენობა ველზე მეტია, რაოდენობა ველის ზომით შემოიფარგლება.

esp_err_t esp_efuse_read_field_cnt(კონსტ esp_efuse_desc_t *ველი[], size_t *გარეთ_ცენტი)¶

კითხულობს ბიტებს EFUSE ველიდან და აბრუნებს „1“-ად დაპროგრამებული ბიტების რაოდენობას.

თუ ბიტები თანმიმდევრულად არ არის დაყენებული, ისინი მაინც დაითვლება.

დაბრუნება
  • ESP_OK: ოპერაცია წარმატებით დასრულდა.
  • ESP_ERR_INVALID_ARG: შეცდომა გადაცემულ არგუმენტებში.
პარამეტრები
  • fieldefuse-ის ველების აღმწერი სტრუქტურის მაჩვენებელი.
  • out_cntმაჩვენებელი, რომელიც შეიცავს „1“ ბიტებად დაპროგრამებული რაოდენობის ბიტებს.

esp_err_t esp_efuse_write_field_blob(კონსტ esp_efuse_desc_t *ველი[], კონსტ void *src, size_t src_ზომის_ბიტები)¶

მასივის EFUSE ველში ჩაწერა.

ჩაწერის ბიტების რაოდენობა შემოიფარგლება „field“ სტრუქტურაში ბიტების აღწერილობიდან მინიმალური მნიშვნელობით ან „src_size_bits“ საჭირო ზომით. ველის სიგრძის დასადგენად გამოიყენეთ „esp_efuse_get_field_size()“ ფუნქცია. ფუნქციის დასრულების შემდეგ, ჩაწერის რეგისტრები გაიწმინდება.

დაბრუნება
  • ESP_OK: ოპერაცია წარმატებით დასრულდა.
  • ESP_ERR_INVALID_ARG: შეცდომა გადაცემულ არგუმენტებში.
  • ESP_ERR_EFUSE_REPEATED_PROG: დაპროგრამებული ბიტების განმეორებითი პროგრამირება მკაცრად აკრძალულია.
  • ESP_ERR_CODING: მონაცემთა შეცდომის დიაპაზონი არ ემთხვევა კოდირების სქემას.
პარამეტრები
  • fieldefuse-ის ველების აღმწერი სტრუქტურის მაჩვენებელი.
  • src: მასივის მიმთითებელი, რომელიც შეიცავს ჩასაწერ მონაცემებს.
  • src_size_bits: ჩასაწერად საჭირო ბიტების რაოდენობა.

esp_err_t esp_efuse_write_field_cnt(კონსტ esp_efuse_desc_t *ველი[], size_t ცნტ)¶

EFUSE ველში საჭირო ბიტების რაოდენობას „1“-ად წერს.

თუ ველში არ არის თავისუფალი ბიტები, რათა საჭირო რაოდენობის ბიტები „1“-ზე დაყენდეს, დაბრუნდება შეცდომა ESP_ERR_EFUSE_CNT_IS_FULL, ველი ნაწილობრივ არ ჩაიწერება. ფუნქციის დასრულების შემდეგ, ჩაწერის რეგისტრები იწმინდება.

დაბრუნება
  • ESP_OK: ოპერაცია წარმატებით დასრულდა.
  • ESP_ERR_INVALID_ARG: შეცდომა გადაცემულ არგუმენტებში.
  • ESP_ERR_EFUSE_CNT_IS_FULL: ყველა მოთხოვნილი cnt ბიტი არ არის დაყენებული.
პარამეტრები
  • fieldefuse-ის ველების აღმწერი სტრუქტურის მაჩვენებელი.
  • cnt: „1“ ბიტებად დაპროგრამებული საჭირო რაოდენობა.

აპლიკაციის სურათის ფორმატი¶

აპლიკაციის სურათი შედგება შემდეგი სტრუქტურებისგან:

  1. ის esp_image_header_t სტრუქტურა აღწერს SPI ფლეშ მეხსიერება ის რეჟიმს და მეხსიერების სეგმენტების რაოდენობას.
  2. ის esp_image_segment_header_t სტრუქტურა აღწერს თითოეულ სეგმენტს, მის სიგრძეს და მდებარეობას ESP32-ის მეხსიერებაში, რასაც მოჰყვება მონაცემები სიგრძით data_lenგამოსახულების თითოეული სეგმენტის მონაცემთა ოფსეტი გამოითვლება შემდეგი გზით:
  • 0 სეგმენტის ოფსეტი = sizeof(esp_image_header_t) + ზომა(esp_image_segment_header_t).
  • ოფსეტი 1 სეგმენტისთვის = ოფსეტი 0 სეგმენტისთვის + 0 სეგმენტის სიგრძე + ზომა(esp_image_segment_header_t).
  • ოფსეტი 2 სეგმენტისთვის = ოფსეტი 1 სეგმენტისთვის + 1 სეგმენტის სიგრძე + ზომა(esp_image_segment_header_t).
  • …

თითოეული სეგმენტის რაოდენობა განისაზღვრება segment_count ველი, რომელიც ინახება esp_image_header_tრაოდენობა არ შეიძლება იყოს მეტი ვიდრე ESP_IMAGE_MAX_SEGMENTS.

მეხსიერების სეგმენტების ტიპისა და მათი მისამართების დიაპაზონების შესახებ დამატებითი ინფორმაციისთვის იხილეთ ESP32 ტექნიკური საცნობარო სახელმძღვანელო, ნაწილი 1.3.2. ჩაშენებული მეხსიერება.

  1. სურათს ბოლო სეგმენტის შემდეგ ერთი საკონტროლო ჯამის ბაიტი აქვს. ეს ბაიტი თექვსმეტბაიტიან შევსებულ საზღვარზეა ჩაწერილი, ამიტომ აპლიკაციის სურათს შესაძლოა შევსება დასჭირდეს.
  2. თუ hash_appended ველი esp_image_header_t დაყენების შემთხვევაში, დაემატება SHA256 ჩეკის ჯამი. SHA256-ის მნიშვნელობა გამოითვლება პირველი ბაიტიდან ამ ველამდე დიაპაზონში. ამ ველის სიგრძე 32 ბაიტია.
  3. თუ ვარიანტები CONFIG_SECURE_SIGNED_APPS_NO_SECURE_BOOT ან CONFIG_SECURE_BOOT_ENABLED თუ ჩართულია, აპლიკაციის გამოსახულებას ექნება დამატებითი 68 ბაიტი ECDSA ხელმოწერისთვის, რაც მოიცავს:
  • ვერსიის სიტყვა (4 ბაიტი),
  • ხელმოწერის მონაცემები (64 ბაიტი).

აპლიკაციის აღწერა¶

ის DROM სეგმენტი იწყება იმით, esp_app_desc_t სტრუქტურა, რომელიც შეიცავს კონკრეტულ ველებს, რომლებიც აღწერენ აპლიკაციას:

  • secure_version - იხილეთ ანტი-rollback.
  • version - იხილეთ აპლიკაციის ვერსია. *
  • project_name ივსება -დან PROJECT_NAME. *
  • time და date - კომპილაცია დრო და თარიღი.
  • idf_ver - ESP-IDF ის ვერსია. *
  • app_elf_sha256 - შეიცავს sha256-ს elf აპლიკაციის ფაილისთვის.

* - მაქსიმალური სიგრძეა 32 სიმბოლო, ნულოვანი დასრულების სიმბოლოს ჩათვლით. მაგალითად, თუ სიგრძე PROJECT_NAME თუ 32 სიმბოლოს გადააჭარბებს, ზედმეტი სიმბოლოები არ იქნება გათვალისწინებული.

ეს სტრუქტურა სასარგებლოა სურათების ატვირთვა ed OTA-ს იდენტიფიკაციისთვის, რადგან მას აქვს ფიქსირებული offset = sizeof(esp_image_header_t) + ზომა(esp_image_segment_header_tროგორც კი მოწყობილობა მიიღებს ამ სტრუქტურის შემცველ პირველ ფრაგმენტს, მას ექნება ყველა ინფორმაცია იმის დასადგენად, უნდა გაგრძელდეს თუ არა განახლება.

აპლიკაციაში მორგებული სტრუქტურის დამატება¶

მომხმარებელს ასევე აქვს შესაძლებლობა, ჰქონდეს მსგავსი სტრუქტურა სურათის დასაწყისთან შედარებით ფიქსირებული ოფსეტით. თქვენს სურათზე საკუთარი სტრუქტურის დასამატებლად შეგიძლიათ გამოიყენოთ შემდეგი ნიმუში:

const __attribute__((section(".rodata_custom_desc"))) esp_custom_app_desc_t custom_app_desc = { ... }

მორგებული სტრუქტურის ოფსეტი არის sizeof(esp_image_header_t) + ზომა(esp_image_segment_header_t) + ზომა(esp_app_desc_t).

იმისათვის, რომ გარანტირებული იყოს, რომ მორგებული სტრუქტურა გამოსახულებაში იქნება განთავსებული, მაშინაც კი, თუ ის არ გამოიყენება, უნდა დაამატოთ:

  • მარკისთვის: დამატება COMPONENT_ADD_LDFLAGS += -u custom_app_desc შიგნით component.mk
  • Cmake-სთვის: დაამატეთ target_link_libraries(${COMPONENT_TARGET} "-u custom_app_desc") შიგნით CMakeLists.txt

uint8_t hash_appended¶

თუ 1, ჩეკის ჯამის შემდეგ ემატება SHA256 დაიჯესტი „მარტივი ჰეში“ (მთელი სურათის). შედის სურათის სიგრძეში. ეს დაიჯესტი ცალკეა უსაფრთხო ჩატვირთვისთვის და გამოიყენება მხოლოდ დაზიანების აღმოსაჩენად. უსაფრთხო ჩატვირთვის ხელმოწერილი სურათებისთვის, ხელმოწერა ემატება ამის შემდეგ (და მარტივი ჰეში შედის ხელმოწერილ მონაცემებში).

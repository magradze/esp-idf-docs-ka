SD/SDIO/MMC დრაივერი¶

[中文]

მიმოხილვა¶

SD/SDIO/MMC დრაივერი ამჟამად მხარს უჭერს SD მეხსიერებას, SDIO ბარათებს და eMMC ჩიპებს. ეს არის პროტოკოლის დონის დრაივერი, რომელიც აგებულია SDMMC-სა და SD SPI ჰოსტის დრაივერებზე.

SDMMC და SD SPI ჰოსტის დრაივერები (დრაივერი/include/driver/sdmmc_host.h) მოგვაწოდეთ API ფუნქციები:

  • ბრძანებების გაგზავნა დაქვემდებარებულ მოწყობილობებზე
  • მონაცემების გაგზავნა და მიღება
  • ავტობუსში შეცდომის პირობების დამუშავება

ინიციალიზაციისა და კონფიგურაციისთვის გამოყენებული ფუნქციებისთვის:

  • SDMMC მასპინძელი, იხ. SDMMC მასპინძელი API
  • SD SPI მასპინძელი, იხ. SD SPI მასპინძელი API

ამ დოკუმენტში აღწერილი SDMMC პროტოკოლის ფენა ამუშავებს SD პროტოკოლის სპეციფიკას, როგორიცაა ბარათის ინიციალიზაცია და მონაცემთა გადაცემის ბრძანებები.

პროტოკოლის დონე მასპინძელთან მუშაობს შემდეგი გზით: sdmmc_host_t სტრუქტურა. ეს სტრუქტურა შეიცავს მასპინძლის სხვადასხვა ფუნქციის მითითებებს.

განაცხადის მაგალითი¶

მაგალითი, რომელიც აერთიანებს SDMMC დრაივერს FATFS ბიბლიოთეკასთან, მოცემულია შემდეგში: მეხსიერება/sd_ბარათი ESP-IDF მაგალითების დირექტორია. ეს მაგალითი ახდენს ბარათის ინიციალიზაციას, შემდეგ იწერს და კითხულობს მონაცემებს მისგან POSIX-ისა და C ბიბლიოთეკის API ის გამოყენებით. დამატებითი ინფორმაციისთვის იხილეთ README.md ფაილი მაგალითის დირექტორიაში.

პროტოკოლის დონე API¶

პროტოკოლის ფენას ენიჭება sdmmc_host_t სტრუქტურა. ეს სტრუქტურა აღწერს SD/MMC მასპინძელ დრაივერს, ჩამოთვლის მის შესაძლებლობებს და იძლევა მითითებებს დრაივერის ფუნქციებზე. პროტოკოლის დონე ინახავს ბარათის სპეციფიკურ ინფორმაციას sdmmc_card_t სტრუქტურა. SD/MMC მასპინძელ დრაივერზე ბრძანებების გაგზავნისას, პროტოკოლის დონე იყენებს sdmmc_command_t სტრუქტურა ბრძანების, არგუმენტების, მოსალოდნელი დაბრუნების მნიშვნელობების და გადასაცემი მონაცემების აღსაწერად, თუ ასეთი არსებობს.

API გამოყენება SD მეხსიერების ბარათებით¶

  1. ჰოსტის ინიციალიზაციისთვის გამოიძახეთ ჰოსტის დრაივერის ფუნქციები, მაგ., sdmmc_host_init(), sdmmc_host_init_slot().
  2. ბარათის ინიციალიზაციისთვის, დარეკეთ sdmmc_card_init() და გადაეცით მას პარამეტრები host - მასპინძელი დრაივერის ინფორმაცია და card - სტრუქტურის მითითება sdmmc_card_t რომელიც შეივსება ბარათის შესახებ ინფორმაციით ფუნქციის დასრულების შემდეგ.
  3. ბარათის სექტორების წასაკითხად და ჩასაწერად გამოიყენეთ sdmmc_read_sectors() და sdmmc_write_sectors() შესაბამისად და გადაეცით მას პარამეტრი card - ბარათის ინფორმაციის სტრუქტურის მითითება.
  4. თუ ბარათი აღარ გამოიყენება, გამოიძახეთ მასპინძელი დრაივერის ფუნქცია - მაგ. sdmmc_host_deinit() - მასპინძელი პერიფერიული მოწყობილობის გამორთვა და დრაივერის მიერ გამოყოფილი რესურსების განთავისუფლება.

API გამოყენება eMMC ჩიპებთან¶

პროტოკოლის დონის პერსპექტივიდან, eMMC მეხსიერების ჩიპები ზუსტად SD მეხსიერების ბარათების მსგავსად იქცევიან. მიუხედავად იმისა, რომ eMMC-ები ჩიპებია და არ აქვთ ბარათის ფორმ-ფაქტორი, SD ბარათების ტერმინოლოგია მაინც შეიძლება გამოყენებულ იქნას eMMC-ზე პროტოკოლის მსგავსების გამო (sdmmc_card_t, sdmmc_card_initგაითვალისწინეთ, რომ eMMC ჩიპების SPI-ზე გამოყენება შეუძლებელია, რაც მათ SD SPI ჰოსტის დრაივერთან შეუთავსებელს ხდის.

eMMC მეხსიერების ინიციალიზაციისა და წაკითხვის/ჩაწერის ოპერაციების შესასრულებლად, მიჰყევით SD ბარათებისთვის წინა განყოფილებაში მითითებულ ნაბიჯებს.

API გამოყენება SDIO ბარათებთან¶

ინიციალიზაციისა და ზონდირების პროცესი იგივეა, რაც SD მეხსიერების ბარათების შემთხვევაში. ერთადერთი განსხვავება SDIO რეჟიმში მონაცემთა გადაცემის ბრძანებებშია.

ბარათის ინიციალიზაციისა და ზონდირების დროს, შესრულებული sdmmc_card_init(), დრაივერი მხოლოდ IO ბარათის შემდეგ რეგისტრებს აკონფიგურირებს:

  1. ბარათის IO ნაწილის გადატვირთვა ხდება I/O Abort (0x06) რეგისტრში RES ბიტის დაყენებით.
  2. თუ ჰოსტისა და სლოტის კონფიგურაციაში ჩართულია 4-ხაზიანი რეჟიმი, დრაივერი ცდილობს დააყენოს ავტობუსის სიგანის ველი ავტობუსის ინტერფეისის კონტროლის (0x07) რეგისტრში. თუ ველის დაყენება წარმატებით დასრულდა, რაც ნიშნავს, რომ მონა მოწყობილობა მხარს უჭერს 4-ხაზიან რეჟიმს, ჰოსტიც გადაერთვება 4-ხაზიან რეჟიმში.
  3. თუ მასპინძლის კონფიგურაციაში ჩართულია მაღალსიჩქარიანი რეჟიმი, SHS ბიტი დაყენებულია მაღალსიჩქარიან (0x13) რეგისტრში.

კერძოდ, დრაივერი არ ადგენს არცერთ ბიტს (1) I/O Enable და Int Enable რეგისტრებში, (2) I/O ბლოკის ზომებში და ა.შ. აპლიკაციებს შეუძლიათ მათი დაყენება გამოძახებით. sdmmc_io_write_byte().

ბარათის კონფიგურაციისა და მონაცემთა გადაცემისთვის, ქვემოთ მოცემული ცხრილიდან აირჩიეთ თქვენი საქმისთვის შესაბამისი ფუნქციების წყვილი.

მოქმედება წაკითხვის ფუნქცია ფუნქციის ჩაწერა
ერთი ბაიტის წაკითხვა და ჩაწერა IO_RW_DIRECT-ის (CMD52) გამოყენებით sdmmc_io_read_byte() sdmmc_io_write_byte()
ბაიტის რეჟიმში IO_RW_EXTENDED (CMD53)-ის გამოყენებით მრავალი ბაიტის წაკითხვა და ჩაწერა sdmmc_io_read_bytes() sdmmc_io_write_bytes()
მონაცემთა ბლოკების წაკითხვა და ჩაწერა IO_RW_EXTENDED (CMD53)-ის გამოყენებით ბლოკურ რეჟიმში sdmmc_io_read_blocks() sdmmc_io_write_blocks()

SDIO შეფერხებების ჩართვა შესაძლებელია აპლიკაციის მიერ ფუნქციის გამოყენებით sdmmc_io_enable_int()SDIO-ს 1-ხაზიან რეჟიმში გამოყენებისას, SDIO შეფერხებების გამოსაყენებლად ასევე საჭიროა D1 ხაზის დაკავშირება.

თუ გსურთ, რომ აპლიკაციამ დაელოდოს SDIO შეფერხებას, გამოიყენეთ sdmmc_io_wait_int().

არსებობს კომპონენტი ESSL (ESP სერიული მონა-კავშირი), რომელიც უნდა გამოიყენოთ, თუ ESP32 SDIO მონა-კავშირთან ურთიერთობთ. იხილეთ ESP სერიული მონა ბმული და მაგალითი პერიფერიული მოწყობილობები/sdio/ჰოსტი.

კომბინირებული (მეხსიერება + IO) ბარათები¶

დრაივერი არ უჭერს მხარს SD კომბო ბარათებს. კომბო ბარათები განიხილება, როგორც IO ბარათები.

ძაფის უსაფრთხოება¶

აპლიკაციების უმეტესობას პროტოკოლის ფენის გამოყენება მხოლოდ ერთ დავალებაში სჭირდება. ამ მიზეზით, პროტოკოლის ფენა არ ახორციელებს რაიმე სახის დაბლოკვას. sdmmc_card_t სტრუქტურაზე, ან SDMMC ან SD SPI ჰოსტის დრაივერებზე წვდომისას. ასეთი დაბლოკვა, როგორც წესი, უფრო მაღალ დონეზე ხორციელდება, მაგალითად, ფაილური სისტემის დრაივერში.

შესაძლოა, თქვენ მიუთითოთ ბუფერი, რომელიც არ არის საკმარისი CIS მონაცემების შესანახად. ამ შემთხვევაში, ეს ფუნქცია თქვენს ბუფერში ინახავს რაც შეიძლება მეტ მონაცემს. ასევე, ეს ფუნქცია შეეცდება მიიღოს და დააბრუნოს თქვენთვის საჭირო ზომა.

  • ESP_ERR_INVALID_SIZE: CIS_CODE_END მოიძებნა, თუმცა buffer_size საჭირო ზომაზე ნაკლებია, რომელიც შემდეგ inout_cis_size-ში ინახება.
  • ESP_ERR_NOT_FOUND: თუ CIS_CODE_END ვერ მოიძებნა. გაზარდეთ inout_cis_size-ის შეყვანის მნიშვნელობა ან დააყენეთ 0-ზე, თუ მაინც გსურთ დასასრულის ძებნა; inout_cis_size-ის გამომავალი მნიშვნელობა ამ შემთხვევაში არასწორია.
  • და სხვა შეცდომის კოდი, რომელიც sdmmc_io_read_bytes-დან დაბრუნდება

  • out_bufferCIS მონაცემების გამომავალი ბუფერი
  • buffer_size: ბუფერის ზომა.
  • inout_cis_sizeსავალდებულო, ზომაზე, შეყვანაზე და გამომავალზე მითითება.
    • შეყვანა: მაქსიმალური ძიების დიაპაზონის ლიმიტი უნდა იყოს 0 ან მეტი, ვიდრე buffer_size. ფუნქცია ეძებს CIS_CODE_END-ს ამ დიაპაზონამდე. დააყენეთ 0-ზე უსასრულო ძიებისთვის.
    • გამომავალი: ყველა CIS მონაცემის შესანახად საჭირო ზომა, თუ მოიძებნება CIS_CODE_END.

esp_err_t sdmmc_io_print_cis_info(uint8_t *ბუფერი, size_t ბუფერის ზომა, FILE *FP)¶

SDIO ბარათის CIS ინფორმაციის გაანალიზება და დაბეჭდვა.

შენიშვნა
Not all the CIS codes and all kinds of tuples are supported. If you see some unresolved code, you can add the parsing of these code in sdmmc_io.c and contribute to the IDF through the Github repository. 
         using sdmmc_card_init
დაბრუნება
  • ESP_OK: წარმატების შემთხვევაში
  • ESP_ERR_NOT_SUPPORTED: თუ ბარათიდან მიღებული მნიშვნელობის დამუშავება მხარდაჭერილი არ არის.
  • ESP_ERR_INVALID_SIZE: თუ CIS ზომის ველები არასწორია.
პარამეტრები
  • buffer: ბუფერის დამუშავება
  • buffer_size: ბუფერის ზომა.
  • fp: ფაილის მაჩვენებელი დასაბეჭდად, stdout-ში დასაბეჭდად დაყენებულია NULL-ზე.

ESP-MESH პროგრამირების სახელმძღვანელო¶

ეს არის ESP-MESH-ის პროგრამირების სახელმძღვანელო, რომელიც მოიცავს API საცნობარო ინსტრუქციებსა და კოდირების მაგალითებს. სახელმძღვანელო დაყოფილია შემდეგ ნაწილებად:

  1. ESP-MESH პროგრამირების მოდელი
  2. ESP-MESH აპლიკაციის დაწერა
  3. თვითორგანიზებული ქსელური მუშაობა
  4. გამოყენების მაგალითები
  5. API მითითება

ESP-MESH პროტოკოლის შესახებ დოკუმენტაციისთვის, გთხოვთ, იხილოთ ESP-MESH API სახელმძღვანელოESP-MESH-ის განვითარების ჩარჩოს შესახებ დამატებითი ინფორმაციისთვის იხილეთ ESP-MESH-ის განვითარების ჩარჩო.

ESP-MESH პროგრამირების მოდელი¶

პროგრამული უზრუნველყოფის დასტა¶

ESP-MESH პროგრამული უზრუნველყოფის დასტა აგებულია Wi-Fi Driver/Free RTOS თავზე და ზოგიერთ შემთხვევაში შეიძლება გამოიყენოს LwIP დასტა (ანუ root node). შემდეგი დიაგრამა ასახავს ESP-MESH პროგრამული უზრუნველყოფის დასტას.

ESP-MESH პროგრამული უზრუნველყოფის დასტა

სისტემის მოვლენები¶

აპლიკაცია ESP-MESH-თან ურთიერთქმედებს შემდეგი გზით: ESP-MESH ღონისძიებებირადგან ESP-MESH აგებულია Wi-Fi დასტის თავზე, აპლიკაციისთვის ასევე შესაძლებელია Wi-Fi დრაივერთან ურთიერთქმედება შემდეგი გზით: Wi-Fi მოვლენის დავალებაშემდეგი დიაგრამა ასახავს ESP-MESH აპლიკაციაში სხვადასხვა სისტემური მოვლენების ინტერფეისებს.

ESP-MESH სისტემის მოვლენების მიწოდება

ის mesh_event_id_t განსაზღვრავს ყველა შესაძლო ESP-MESH მოვლენას და შეუძლია მიუთითოს ისეთი მოვლენები, როგორიცაა მშობლის/შვილის კავშირი/გათიშვა. ESP-MESH მოვლენების გამოყენებამდე, აპლიკაციამ უნდა დაარეგისტრიროს Mesh მოვლენების დამმუშავებელი გავლით esp_event_handler_register() ნაგულისხმევ მოვლენის დავალებაში. რეგისტრირებული Mesh მოვლენების დამმუშავებელი შეიცავს აპლიკაციისთვის შესაბამისი თითოეული ESP-MESH მოვლენის დამმუშავებლებს.

mesh მოვლენების ტიპური გამოყენების შემთხვევები მოიცავს ისეთი მოვლენების გამოყენებას, როგორიცაა MESH_EVENT_PARENT_CONNECTED და MESH_EVENT_CHILD_CONNECTED იმის მითითებისთვის, თუ როდის შეუძლია კვანძს მონაცემების გადაცემის დაწყება შესაბამისად ზემოთ და ქვემოთ. ანალოგიურად, IP_EVENT_STA_GOT_IP და IP_EVENT_STA_LOST_IP შეიძლება გამოყენებულ იქნას იმის მითითებისთვის, თუ როდის შეუძლია და როდის არ შეუძლია root კვანძს მონაცემების გადაცემა გარე IP ქსელში.

გაფრთხილება

ESP-MESH-ის თვითორგანიზების რეჟიმში გამოყენებისას, მომხმარებლებმა უნდა დარწმუნდნენ, რომ Wi-Fi API ზე ზარები არ ხორციელდება. ეს იმით არის განპირობებული, რომ თვითორგანიზების რეჟიმი შიდა რეჟიმში განახორციელებს Wi-Fi API ზარებს დასაკავშირებლად/გათიშვისთვის/სკანირებისთვის და ა.შ. აპლიკაციიდან ნებისმიერი Wi-Fi გამოძახება (მათ შორის უკუკავშირიდან გამოძახებები და Wi-Fi მოვლენების დამმუშავებლები) შეიძლება ხელს უშლიდეს ESP-MESH-ის თვითორგანიზებულ ქცევას.ამიტომ, მომხმარებლებმა არ უნდა გამოიძახონ Wi-Fi API s-ის შემდეგ esp_mesh_start() ჰქვია და მანამდე esp_mesh_stop() ეწოდება.

LwIP და ESP-MESH¶

აპლიკაციას შეუძლია ESP-MESH სტეკზე პირდაპირ წვდომა LwIP სტეკის გავლის გარეშე. LwIP სტეკი root კვანძს მხოლოდ გარე IP ქსელიდან მონაცემების გადასაცემად/მისაღებად სჭირდება. თუმცა, რადგან ყველა კვანძს შეუძლია გახდეს root კვანძი (root კვანძის ავტომატური შერჩევის გამო), თითოეულმა კვანძმა მაინც უნდა მოახდინოს LwIP სტეკის ინიციალიზაცია.

თითოეულ კვანძს მოეთხოვება LwIP-ის ინიციალიზაცია გამოძახებით. tcpip_adapter_init()LwIP-ზე არა-root კვანძის წვდომის თავიდან ასაცილებლად, აპლიკაციამ LwIP ინიციალიზაციის შემდეგ უნდა შეწყვიტოს შემდეგი სერვისები:

  • DHCP სერვერის სერვისი softAP ინტერფეისზე.
  • DHCP კლიენტის სერვისი სადგურის ინტერფეისზე.

შემდეგი კოდის ფრაგმენტი აჩვენებს, თუ როგორ უნდა მოხდეს LwIP-ის ინიციალიზაცია ESP-MESH აპლიკაციებისთვის.

/*  tcpip initialization */
tcpip_adapter_init();
/*
 * for mesh
 * stop DHCP server on softAP interface by default
 * stop DHCP client on station interface by default
 */
ESP_ERROR_CHECK(tcpip_adapter_dhcps_stop(TCPIP_ADAPTER_IF_AP));
ESP_ERROR_CHECK(tcpip_adapter_dhcpc_stop(TCPIP_ADAPTER_IF_STA));

შენიშვნა

ESP-MESH-ისთვის საჭიროა root კვანძის როუტერთან დაკავშირება. ამიტომ, იმ შემთხვევაში, თუ კვანძი root გახდება, შესაბამისმა დამმუშავებელმა უნდა გაუშვას DHCP კლიენტის სერვისი და დაუყოვნებლივ მიიღოს IP მისამართიეს სხვა კვანძებს საშუალებას მისცემს დაიწყონ პაკეტების გადაცემა/მიღება გარე IP ქსელიდან/ქსელიდან. თუმცა, ეს ნაბიჯი არ არის საჭირო, თუ გამოიყენება სტატიკური IP პარამეტრები.

ESP-MESH აპლიკაციის დაწერა¶

ESP-MESH-ის გაშვების წინაპირობაა LwIP-ის და Wi-Fi ინიციალიზაცია. შემდეგი კოდის ფრაგმენტი აჩვენებს აუცილებელ წინაპირობას, სანამ თავად ESP-MESH-ის ინიციალიზაცია მოხდება.

tcpip_adapter_init();
/*
 * for mesh
 * stop DHCP server on softAP interface by default
 * stop DHCP client on station interface by default
 */
ESP_ERROR_CHECK(tcpip_adapter_dhcps_stop(TCPIP_ADAPTER_IF_AP));
ESP_ERROR_CHECK(tcpip_adapter_dhcpc_stop(TCPIP_ADAPTER_IF_STA));

/*  event initialization */
ESP_ERROR_CHECK(esp_event_loop_create_default());

/*  Wi-Fi initialization */
wifi_init_config_t config = WIFI_INIT_CONFIG_DEFAULT();
ESP_ERROR_CHECK(esp_wifi_init(&config));
/*  register IP events handler */
ESP_ERROR_CHECK(esp_event_handler_register(IP_EVENT, IP_EVENT_STA_GOT_IP, &ip_event_handler, NULL));
ESP_ERROR_CHECK(esp_wifi_set_storage(WIFI_STORAGE_FLASH));
ESP_ERROR_CHECK(esp_wifi_start());

LwIP-ისა და Wi-Fi ინიციალიზაციის შემდეგ, ESP-MESH ქსელის გაშვების პროცესი შეიძლება შეჯამდეს შემდეგ სამ ეტაპად:

  1. ბადის ინიციალიზაცია
  2. ESP-MESH ქსელის კონფიგურაცია
  3. დაწყება ბადისებრი

ბადის ინიციალიზაცია¶

შემდეგი კოდის ფრაგმენტი აჩვენებს, თუ როგორ უნდა მოხდეს ESP-MESH-ის ინიციალიზაცია

/*  mesh initialization */
ESP_ERROR_CHECK(esp_mesh_init());
/*  register mesh events handler */
ESP_ERROR_CHECK(esp_event_handler_register(MESH_EVENT, ESP_EVENT_ANY_ID, &mesh_event_handler, NULL));

ESP-MESH ქსელის კონფიგურაცია¶

ESP-MESH კონფიგურირებულია esp_mesh_set_config() რომელიც თავის არგუმენტებს იღებს შემდეგი ფორმულის გამოყენებით: mesh_cfg_t სტრუქტურა. სტრუქტურა შეიცავს შემდეგ პარამეტრებს, რომლებიც გამოიყენება ESP-MESH-ის კონფიგურაციისთვის:

პარამეტრი აღწერა
არხი დიაპაზონი 1-დან 14-მდე
ბადისებრი ID ESP-MESH ქსელის ID, იხ. mesh_addr_t
როუტერი როუტერის კონფიგურაცია, იხილეთ mesh_router_t
Mesh AP Mesh AP კონფიგურაცია, იხილეთ mesh_ap_cfg_t
კრიპტო ფუნქციები Mesh IE-ს კრიპტო ფუნქციები, იხილეთ mesh_crypto_funcs_t

შემდეგი კოდის ფრაგმენტი აჩვენებს, თუ როგორ უნდა დააკონფიგურიროთ ESP-MESH.

/* Enable the Mesh IE encryption by default */
mesh_cfg_t cfg = MESH_INIT_CONFIG_DEFAULT();
/* mesh ID */
memcpy((uint8_t *) &cfg.mesh_id, MESH_ID, 6);
/* channel (must match the router's channel) */
cfg.channel = CONFIG_MESH_CHANNEL;
/* router */
cfg.router.ssid_len = strlen(CONFIG_MESH_ROUTER_SSID);
memcpy((uint8_t *) &cfg.router.ssid, CONFIG_MESH_ROUTER_SSID, cfg.router.ssid_len);
memcpy((uint8_t *) &cfg.router.password, CONFIG_MESH_ROUTER_PASSWD,
       strlen(CONFIG_MESH_ROUTER_PASSWD));
/* mesh softAP */
cfg.mesh_ap.max_connection = CONFIG_MESH_AP_CONNECTIONS;
memcpy((uint8_t *) &cfg.mesh_ap.password, CONFIG_MESH_AP_PASSWD,
       strlen(CONFIG_MESH_AP_PASSWD));
ESP_ERROR_CHECK(esp_mesh_set_config(&cfg));

დაწყება ბადისებრი¶

შემდეგი კოდის ფრაგმენტი აჩვენებს, თუ როგორ უნდა დაიწყოთ ESP-MESH.

/* mesh start */
ESP_ERROR_CHECK(esp_mesh_start());

ESP-MESH-ის გაშვების შემდეგ, აპლიკაციამ უნდა შეამოწმოს ESP-MESH მოვლენები, რათა დაადგინოს, როდის დაუკავშირდა ის ქსელს. დაკავშირების შემდეგ, აპლიკაციას შეუძლია დაიწყოს პაკეტების გადაცემა და მიღება ESP-MESH ქსელში. esp_mesh_send() და esp_mesh_recv().

თვითორგანიზებული ქსელური მუშაობა¶

თვითორგანიზებული ქსელური კავშირი ESP-MESH-ის ფუნქციაა, სადაც კვანძებს შეუძლიათ ავტონომიურად სკანირება/შერჩევა/დაკავშირება/ხელახლა დაკავშირება სხვა კვანძებთან და როუტერებთან. ეს ფუნქცია საშუალებას აძლევს ESP-MESH ქსელს იმუშაოს მაღალი ხარისხის ავტონომიით, ქსელის დინამიური ქსელის ტოპოლოგიებისა და პირობების მიმართ მდგრადობის გაზრდით. თვითორგანიზებული ქსელური კავშირის ჩართვის შემთხვევაში, ESP-MESH ქსელის კვანძებს შეუძლიათ შემდეგი მოქმედებების შესრულება ავტონომიურად:

  • ძირეული კვანძის შერჩევა ან არჩევა (იხ. ავტომატური ძირეული კვანძის შერჩევა ში ქსელის შექმნა)
  • სასურველი მშობელი კვანძის შერჩევა (იხ. მშობელი კვანძის შერჩევა ში ქსელის შექმნა)
  • ავტომატური ხელახლა დაკავშირება გათიშვის აღმოჩენისთანავე (იხ. შუალედური მშობელი კვანძის შეცდომა ში ქსელის მართვა)

როდესაც თვითორგანიზებული ქსელი ჩართულია, ESP-MESH სტეკი შიდა ზარებს განახორციელებს Wi-Fi API ზე. ამიტომ, აპლიკაციის დონემ არ უნდა განახორციელოს რაიმე ზარი Wi-Fi API ზე, სანამ თვითორგანიზებული ქსელი ჩართულია, რადგან ამან შეიძლება გამოიწვიოს ESP-MESH-თან ჩარევის რისკი..

თვითორგანიზებული ქსელების გამორთვა¶

თვითორგანიზებული ქსელური ფუნქციონირების ჩართვა ან გამორთვა აპლიკაციის მიერ გაშვების დროს შესაძლებელია შემდეგი ფუნქციის გამოძახებით: esp_mesh_set_self_organized() ფუნქცია. ფუნქციას აქვს ორი შემდეგი პარამეტრი:

  • bool enable განსაზღვრავს, ჩართოს თუ გამორთოს თვითორგანიზებული ქსელი.
  • bool select_parent განსაზღვრავს, უნდა შეირჩეს თუ არა ახალი მშობელი კვანძი თვითორგანიზებული ქსელის ჩართვისას. ახალი მშობლის არჩევას განსხვავებული ეფექტი აქვს კვანძის ტიპისა და კვანძის მიმდინარე მდგომარეობის მიხედვით. ეს პარამეტრი არ გამოიყენება თვითორგანიზებული ქსელის გამორთვისას.

თვითორგანიზებული ქსელური მუშაობის გამორთვა¶

შემდეგი კოდის ფრაგმენტი აჩვენებს, თუ როგორ გამორთოთ თვითორგანიზებული ქსელი.

//Disable self organized networking
esp_mesh_set_self_organized(false, false);

ESP-MESH შეეცდება შეინარჩუნოს კვანძის მიმდინარე Wi-Fi მდგომარეობა თვითორგანიზებული ქსელის გამორთვისას.

  • თუ კვანძი ადრე სხვა კვანძებთან იყო დაკავშირებული, ის დაკავშირებული დარჩება.
  • თუ კვანძი ადრე გათიშული იყო და მშობელი კვანძის ან როუტერის სკანირებას ახდენდა, ის შეწყვეტს სკანირებას.
  • თუ კვანძი ადრე ცდილობდა მშობელ კვანძთან ან როუტერთან ხელახლა დაკავშირებას, ის შეწყვეტს ხელახლა დაკავშირებას.

თვითორგანიზებული ქსელური მუშაობის ხელშეწყობა¶

ESP-MESH შეეცდება შეინარჩუნოს კვანძის მიმდინარე Wi-Fi მდგომარეობა თვითორგანიზებული ქსელის ჩართვისას. თუმცა, კვანძის ტიპისა და ახალი მშობლის არჩევის მიხედვით, კვანძის Wi-Fi მდგომარეობა შეიძლება შეიცვალოს. ქვემოთ მოცემულ ცხრილში ნაჩვენებია თვითორგანიზებული ქსელის ჩართვის ეფექტები.

მშობლის არჩევა ძირეული კვანძია ეფექტები
ჩრ ჩრ
  • მშობელ კვანძთან უკვე დაკავშირებული კვანძები დაკავშირებულად დარჩებიან.
  • კვანძები, რომლებიც ადრე სკანირებდნენ მშობელ კვანძს, შეწყვეტენ სკანირებას. გამოძახება. esp_mesh_connect() გადატვირთვისთვის.
Y
  • როუტერთან უკვე დაკავშირებული root კვანძი დაკავშირებული დარჩება.
  • როუტერიდან გათიშულ root კვანძს დასჭირდება გამოძახება esp_mesh_connect() ხელახლა დასაკავშირებლად.

  • მშობელ კვანძთან უკვე დაკავშირებული კვანძები გაითიშება, ხელახლა აირჩევს სასურველ მშობელ კვანძს და დაუკავშირდება.

  • იმისათვის, რომ root კვანძი დაუკავშირდეს მშობელ კვანძს, მან უნდა უარი თქვას root-ის როლზე. შესაბამისად, root კვანძი გაითიშება როუტერიდან და ყველა შვილობილი კვანძიდან, შეარჩევს სასურველ მშობელ კვანძს და დაუკავშირდება.

შემდეგი კოდის სნიპინგში ნაჩვენებია, თუ როგორ უნდა ჩართოთ თვითორგანიზებული ქსელი.

//Enable self organized networking and select a new parent
esp_mesh_set_self_organized(true, true);

...

//Enable self organized networking and manually reconnect

Wi-Fi გამოძახება API¶

შეიძლება არსებობდეს შემთხვევები, როდესაც აპლიკაციას შეიძლება სურდეს Wi-Fi API პირდაპირ გამოძახება ESP-MESH-ის გამოყენებისას. მაგალითად, აპლიკაციას შეიძლება სურდეს მეზობელი წვდომის წერტილების ხელით სკანირება. თუმცა, თვითორგანიზებული ქსელი უნდა გამორთოთ აპლიკაციის მიერ Wi-Fi API s-ის გამოძახებამდე.ეს ხელს შეუშლის ESP-MESH დასტას Wi-Fi და API ის გამოძახების მცდელობაში და პოტენციურად ხელს შეუშლის აპლიკაციის გამოძახებებს.

ამიტომ, Wi-Fi და API ის აპლიკაციის გამოძახებები უნდა განთავსდეს შემდეგის გამოძახებებს შორის. esp_mesh_set_self_organized() რომელიც გამორთავს და ჩართავს თვითორგანიზებულ ქსელს. შემდეგი კოდის ფრაგმენტი აჩვენებს, თუ როგორ შეუძლია აპლიკაციას უსაფრთხოდ გამოიძახოს esp_wifi_scan_start() ESP-MESH-ის გამოყენებისას.

//Disable self organized networking
esp_mesh_set_self_organized(0, 0);

//Stop any scans already in progress
esp_wifi_scan_stop();
//Manually start scan. Will automatically stop when run to completion
esp_wifi_scan_start();

//Process scan results

...

//Re-enable self organized networking if still connected
esp_mesh_set_self_organized(1, 0);

...

//Re-enable self organized networking if non-root and disconnected
esp_mesh_set_self_organized(1, 1);

...

//Re-enable self organized networking if root and disconnected
esp_mesh_set_self_organized(1, 0);  //Don't select new parent
esp_mesh_connect();                 //Manually reconnect to router

გამოყენების მაგალითები¶

ESP-IDF შეიცავს შემდეგ ESP-MESH პროექტებს:

შიდა კომუნიკაციის მაგალითი აჩვენებს, თუ როგორ უნდა დააყენოთ ESP-MESH ქსელი და როგორ დაავალოთ root კვანძს მონაცემთა პაკეტის გაგზავნა ქსელის ყველა კვანძისთვის.

ხელით ქსელის შექმნის მაგალითი აჩვენებს, თუ როგორ გამოვიყენოთ ESP-MESH თვითორგანიზების ფუნქციების გარეშე. ეს მაგალითი გვიჩვენებს, თუ როგორ დავპროგრამოთ კვანძი ისე, რომ ხელით დაასკანიროს პოტენციური მშობელი კვანძების სია და აირჩიოს მშობელი კვანძი მორგებული კრიტერიუმების საფუძველზე.

    • მიუთითეთ, ეს API ბლოკურია თუ არაბლოკური, ბლოკი ნაგულისხმევად
      • თუ საჭიროა არაბლოკირება, უნდა დაყენდეს MESH_DATA_NONBLOCK.
    • root-ის შეცვლის შემთხვევაში, MESH_DATA_DROP განსაზღვრავს, რომ ეს პაკეტი შეიძლება გადაეცეს ახალ root-ს გარე IP ქსელში მონაცემების გადასაცემად. ჩვენ ვცდილობთ თავიდან ავიცილოთ root-ის შეცვლით გამოწვეული მონაცემების დაკარგვა, მაგრამ არსებობს რისკი, რომ ახალ root-ს მეხსიერება ამოეწუროს, რადგან მეხსიერების უმეტესი ნაწილი დაკავებულია მომლოდინე მონაცემებით, რომლებიც esp_mesh_recv_toDS()-ის მიერ დროულად არ იკითხება.

      ზოგადად, ჩვენ ვთავაზობთ esp_mesh_recv_toDS() ფუნქციის გამოძახებას IP ქსელთან კავშირის შექმნის შემდეგ. ამგვარად, მონაცემები გარე IP ქსელში სოკეტის საშუალებით მხოლოდ esp_mesh_recv_toDS() ფუნქციის წაკითხვით ხდება, რაც მეხსიერების არასაჭირო კოპირებას თავიდან აგაცილებთ.

  • opt: პარამეტრები

esp_err_t esp_mesh_recv_toDS(mesh_addr_t *-დან, mesh_addr_t *რომ, mesh_data_t *მონაცემები, int დროის ამოწურვა_წმ, int *დროშა, mesh_opt_t არჩევა[], int opt_count)¶

მიიღეთ პაკეტი, რომელიც მიმართულია გარე IP ქსელზე.

  • Root იყენებს ამ API გარე IP ქსელისთვის განკუთვნილი პაკეტების მისაღებად.
  • Root მიღებულ პაკეტებს სოკეტის საშუალებით საბოლოო დანიშნულების ადგილამდე გადააგზავნის.
  • თუ მიღებული პაკეტების გასაგზავნად არცერთი სოკეტის კავშირი არ არის მზად და აპლიკაციებმა არ გამოიძახეს ეს esp_mesh_recv_toDS(), მთელი mesh ქსელიდან მიღებული პაკეტები toDS რიგში მოლოდინში დარჩება.

გამოიყენეთ esp_mesh_get_rx_pending() ფუნქცია, რათა შეამოწმოთ რიგში არსებული პაკეტების რაოდენობა, რომლებიც აპლიკაციების მიერ მიღებას ელოდება root-ში მეხსიერების ამოწურვის შემთხვევაში.

esp_err_t esp_mesh_set_config(კონსტ mesh_cfg_t *კონფიგურაცია)¶

mesh stack-ის კონფიგურაციის დაყენება.

  • ნაგულისხმევი მნიშვნელობების ინიციალიზაციისთვის გამოიყენეთ MESH_INIT_CONFIG_DEFAULT() ფუნქცია, mesh IE ნაგულისხმევად დაშიფრულია.
  • ბადისებრი ქსელი დამყარებულია ფიქსირებულ არხზე (1-14).
  • Mesh მოვლენის უკუგამოძახება სავალდებულოა.
  • Mesh ID არის MBSS-ის იდენტიფიკატორი. ერთი და იგივე mesh ID-ის მქონე კვანძებს შეუძლიათ ერთმანეთთან კომუნიკაცია.
  • როუტერის კონფიგურაციასთან დაკავშირებით, თუ როუტერი დამალულია, BSSID ველი სავალდებულოა.

თუ BSSID ველი არ არის დაყენებული და არსებობს ერთზე მეტი როუტერი ერთი და იგივე SSID-ით, არსებობს რისკი, რომ გამოჩნდეს ერთზე მეტი root, რომლებიც დაკავშირებულია სხვადასხვა BSSID-ით. ეს ნიშნავს, რომ ერთზე მეტი mesh ქსელია შექმნილი ერთი და იგივე mesh ID-ით.

Root კონფლიქტის ფუნქციას შეუძლია აღმოფხვრას ერთი და იგივე BSSID-თან დაკავშირებული ზედმეტი root-ები, მაგრამ ვერ ამუშავებს სხვადასხვა BSSID-თან დაკავშირებულ root-ებს. რადგან მომხმარებლებს შეიძლება ჰქონდეთ იგივე SSID-ით როუტერების დაყენების მოთხოვნები მომავალი ჩანაცვლებისთვის. თუმცა, თუ ზემოთ ჩამოთვლილი სიტუაციები მოხდება, გთხოვთ, დარწმუნდეთ, რომ აპლიკაციები root-ზე ახორციელებენ გადამისამართების ფუნქციებს, რათა უზრუნველყონ სხვადასხვა mesh ქსელში არსებული მოწყობილობების ერთმანეთთან კომუნიკაცია. mesh softAP-ის max_connection შეზღუდულია მხარდაჭერილი Wi-Fi softAP-ების მაქსიმალური რაოდენობით (მაქს: 10).

MESH_EVENT_ROOT_ASKED_YIELD¶

root-ს უფრო ძლიერი არსებული root სთხოვს yield-ის გამოყენებას. თუ თვითორგანიზება გამორთულია და მომხმარებლები ამ მოწყობილობას root-ად განსაზღვრავენ, მომხმარებლებმა ამ მოწყობილობისთვის ახალი მშობელი უნდა დააყენონ. თუ თვითორგანიზება ჩართულია, ეს მოწყობილობა თავად იპოვის ახალ მშობელს, მომხმარებლებს შეუძლიათ ამ მოვლენის იგნორირება.

MESH_EVENT_ROOT_FIXED¶

MESH_EVENT_FIND_NETWORK¶

MESH_EVENT_ROUTER_SWITCH¶

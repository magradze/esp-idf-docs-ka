ESP-NETIF-ის მორგებული შემავალი/გამომავალი დრაივერი¶

ეს განყოფილება აღწერს ახალი შეყვანის/გამოყვანის დრაივერის დანერგვას esp-netif დაკავშირების შესაძლებლობებით. კონვენციის მიხედვით, შეყვანის/გამოყვანის დრაივერმა უნდა დარეგისტრირდეს, როგორც esp-netif დრაივერი და, შესაბამისად, შეინარჩუნოს დამოკიდებულება esp-netif კომპონენტი ზე და პასუხისმგებელია მონაცემთა გზის ფუნქციების, მიმაგრების შემდგომი უკუკავშირის და, უმეტეს შემთხვევაში, ასევე ნაგულისხმევი მოვლენების დამმუშავებლების უზრუნველყოფაზე, რათა განისაზღვროს ქსელური ინტერფეისის მოქმედებები დრაივერის სასიცოცხლო ციკლის გადასვლების საფუძველზე.

პაკეტის შეყვანა/გამოყვანა¶

როგორც დიაგრამაზეა ნაჩვენები, esp-netif-თან დასაკავშირებლად უნდა განისაზღვროს პაკეტის მონაცემთა ბილიკის შემდეგი სამი API ფუნქცია:

  • esp_netif_transmit()
  • esp_netif_free_rx_buffer()
  • esp_netif_receive()

პირველი ორი ფუნქცია გადაცემისა და rx ბუფერის გათავისუფლებისთვის მოცემულია უკუკავშირის სახით, ანუ ისინი გამოიძახება esp-netif-დან (და მისი ძირითადი TCP/IP დასტიდან) და I/O დრაივერი უზრუნველყოფს მათ იმპლემენტაციას.

მეორეს მხრივ, მიმღები ფუნქცია გამოიძახება შემავალი/გამომავალი დრაივერიდან, ამიტომ დრაივერის კოდი უბრალოდ გამოიძახებს esp_netif_receive() მიღებული ახალი მონაცემების მოვლენაზე.

გამოქვეყნება დართული უკუკავშირის ზარი¶

ქსელური ინტერფეისის ინიციალიზაციის ბოლო ნაწილია esp-netif ინსტანციის შეყვანის/გამოყვანის დრაივერთან მიმაგრება შემდეგი API გამოძახებით:

esp_err_t esp_netif_attach(esp_netif_t *esp_netif, esp_netif_iodriver_handle driver_handle);

ვარაუდობენ, რომ esp_netif_iodriver_handle არის დრაივერის ობიექტის მაჩვენებელი, სტრუქტურა, რომელიც მიღებულია struct esp_netif_driver_base_s, ამიტომ I/O დრაივერის სტრუქტურის პირველი წევრი უნდა იყოს ეს საბაზისო სტრუქტურა მითითებებით

  • მიმაგრების შემდგომი ფუნქციის უკუკავშირი
  • დაკავშირებული esp-netif ეგზემპლარი

შედეგად, I/O დრაივერმა უნდა შექმნას ქვემოთ მოცემული სტრუქტურის ეგზემპლარი:

typedef struct my_netif_driver_s {
        esp_netif_driver_base_t base;           /*!< base structure reserved as esp-netif driver */
        driver_impl             *h;             /*!< handle of driver implementation */
    } my_netif_driver_t;

რეალური მნიშვნელობებით my_netif_driver_t::base.post_attach და ფაქტობრივი მძღოლები ამუშავებენ my_netif_driver_t::hასე რომ, როდესაც esp_netif_attach() ინიციალიზაციის კოდიდან გამოძახების შემთხვევაში, შეყვანის/გამოყვანის დრაივერის კოდიდან მიმაგრების შემდგომი უკუკავშირი სრულდება esp-netif-სა და შეყვანის/გამოყვანის დრაივერის ინსტანციებს შორის უკუკავშირების ორმხრივი რეგისტრაციისთვის. როგორც წესი, დრაივერი ასევე ირთვება მიმაგრების შემდგომი უკუკავშირის დროს. ქვემოთ მოცემულია მარტივი მიმაგრების შემდგომი უკუკავშირის მაგალითი:

static esp_err_t my_post_attach_start(esp_netif_t * esp_netif, void * args)
{
    my_netif_driver_t *driver = args;
    const esp_netif_driver_ifconfig_t driver_ifconfig = {
            .driver_free_rx_buffer = my_free_rx_buf,
            .transmit = my_transmit,
            .handle = driver->driver_impl
    };
    driver->base.netif = esp_netif;
    ESP_ERROR_CHECK(esp_netif_set_driver_config(esp_netif, &driver_ifconfig));
    my_driver_start(driver->driver_impl);
    return ESP_OK;
}

ნაგულისხმევი დამმუშავებლები¶

შემავალი/გამომავალი დრაივერები, როგორც წესი, ასევე იძლევიან დაკავშირებული ქსელური ინტერფეისების სასიცოცხლო ციკლის ქცევის ნაგულისხმევ განმარტებებს შემავალი/გამომავალი დრაივერების მდგომარეობის გადასვლებზე დაყრდნობით. მაგალითად. მძღოლის დაწყება -> ქსელის დაწყებადა ა.შ. ასეთი ნაგულისხმევი დამმუშავებლის მაგალითი ქვემოთ არის მოცემული:

esp_err_t my_driver_netif_set_default_handlers(my_netif_driver_t *driver, esp_netif_t * esp_netif)
{
    driver_set_event_handler(driver->driver_impl, esp_netif_action_start, MY_DRV_EVENT_START, esp_netif);
    driver_set_event_handler(driver->driver_impl, esp_netif_action_stop, MY_DRV_EVENT_STOP, esp_netif);
    return ESP_OK;
}

ქსელის დასტის კავშირი¶

პაკეტური მონაცემების გზის ფუნქციები, რომლებიც განკუთვნილია rx ბუფერის გადაცემისა და გათავისუფლებისთვის (განსაზღვრულია I/O დრაივერში), გამოიძახება esp-netif-დან, კერძოდ, მისი TCP/IP დასტის დამაკავშირებელი ფენიდან. შემდეგი API მითითება ასახავს ქსელის დასტის ურთიერთქმედებას esp-netif-თან.

სათაურის ფაილი¶

  • esp_netif/include/esp_netif_net_stack.h

გაითვალისწინეთ, რომ PPP netif impl მაჩვენებლის მოპოვება არ არის მხარდაჭერილი და ეს ფუნქცია დააბრუნებს NULL-ს esp_netif ინსტანციებისთვის, რომლებიც კონფიგურირებულია PPP რეჟიმში.

esp_err_t esp_netif_transmit(esp_netif_t *esp_netif, void *მონაცემები, size_t ლენ)¶

გამოაქვს პაკეტები TCP/IP დასტიდან გადასაცემად განკუთვნილ მედიაში.

ეს ფუნქცია გამოიძახება ქსელის დასტიდან IO დრაივერისთვის პაკეტების გასაგზავნად.

დაბრუნება
ESP_OK on success, an error passed from the I/O driver otherwise
პარამეტრები
  • esp_netif: esp-netif ინსტანციის სახელური
  • dataგადასაცემი მონაცემები
  • lenმონაცემთა ჩარჩოს სიგრძე

void esp_netif_free_rx_buffer(void *esp_netif, void *ბუფერი)¶

გაათავისუფლეთ მედია დრაივერის მიერ გამოყოფილი RX ბუფერი.

ეს ფუნქცია გამოიძახება ქსელის დასტიდან, როდესაც IO დრაივერის კონტექსტში უნდა გათავისუფლდეს rx ბუფერი, ანუ io დრაივერის საკუთრებაში არსებული ბუფერის დეალოქაციისთვის (როდესაც მონაცემთა პაკეტები გადაეცა უფრო მაღალ დონეებს კოპირების თავიდან ასაცილებლად).

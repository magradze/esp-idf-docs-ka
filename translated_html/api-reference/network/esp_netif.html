ESP-NETIF¶

ESP-NETIF ბიბლიოთეკის დანიშნულება ორმაგია:

  • ის უზრუნველყოფს აბსტრაქციის ფენას აპლიკაციისთვის TCP/IP დასტის თავზე. ეს საშუალებას მისცემს აპლიკაციებს მომავალში აირჩიონ IP დასტებს შორის.
  • მის მიერ მოწოდებული API ფაილები ძაფებისთვის უსაფრთხოა, მაშინაც კი, თუ ძირითადი TCP/IP დასტა API ფაილები არ არის უსაფრთხო.

ESP-IDF ამჟამად ESP-NETIF-ს მხოლოდ lwIP TCP/IP დაკისთვის ახორციელებს. თუმცა, თავად ადაპტერი TCP/IP იმპლემენტაციისგან აგნოსტიკურად არის განწყობილი და შესაძლებელია სხვადასხვა იმპლემენტაცია.

ზოგიერთი ESP-NETIF API ფუნქცია განკუთვნილია აპლიკაციის კოდით გამოსაძახებლად, მაგალითად, ინტერფეისის IP მისამართების მისაღებად/დასაყენებლად, DHCP-ის კონფიგურაციისთვის. სხვა ფუნქციები განკუთვნილია ქსელის დრაივერის დონის მიერ ESP-IDF შიდა გამოყენებისთვის.

ბევრ შემთხვევაში, აპლიკაციებს არ სჭირდებათ ESP-NETIF API ის პირდაპირ გამოძახება, რადგან ისინი გამოიძახება ქსელის ნაგულისხმევი მოვლენების დამმუშავებლებისგან.

ESP-NETIF კომპონენტი არის tcpip_adapter-ის, ყოფილი ქსელური ინტერფეისის აბსტრაქციის, მემკვიდრე, რომელიც IDF v4.1-ის შემდეგ მოძველდა. გთხოვთ იხილოთ TCP/IP ადაპტერის მიგრაციის სახელმძღვანელო განყოფილება იმ შემთხვევაში, თუ არსებული აპლიკაციები პორტირებული იქნება esp-netif API გამოსაყენებლად.

ESP-NETIF ურთიერთქმედება¶

ა) მომხმარებლის კოდი, ქვაბის ფირფიტა¶

საკომუნიკაციო მედიისა და კონფიგურირებული TCP/IP ქსელის დასტის კონკრეტულ IO დრაივერთან აპლიკაციის საერთო ურთიერთქმედება აბსტრაგირებულია ESP-NETIF API ის გამოყენებით და აღწერილია ქვემოთ:

  1. ინიციალიზაციის კოდი
  1. IO დრაივერის ინიციალიზაცია
  2. ქმნის ESP-NETIF-ის ახალ ინსტანციას და ახდენს კონფიგურაციას
  • ESP-NETIF-ის სპეციფიკური პარამეტრები (დროშები, ქცევა, სახელი)
  • ქსელის დასტის პარამეტრები (netif init და შეყვანის ფუნქციები, საჯაროდ ხელმისაწვდომი არ არის)
  • IO დრაივერის სპეციფიკური პარამეტრები (გადაცემა, უფასო rx ბუფერული ფუნქციები, IO დრაივერის სახელური)
  1. IO დრაივერის სახელურს ამაგრებს ზემოთ მოცემულ ნაბიჯებში შექმნილ ESP-NETIF ინსტანციაზე.
  2. ახდენს მოვლენების დამმუშავებლების კონფიგურაციას
  • გამოიყენეთ ნაგულისხმევი დამმუშავებლები IO დრაივერებში განსაზღვრული საერთო ინტერფეისებისთვის; ან განსაზღვრეთ კონკრეტული დამმუშავებლები მორგებული ქცევისთვის/ახალი ინტერფეისებისთვის.
  • აპლიკაციასთან დაკავშირებული მოვლენების (მაგალითად, IP-ის დაკარგვის/შეძენის) დამმუშავებლების რეგისტრაცია
  1. ქსელურ ინტერფეისებთან ურთიერთქმედება ESP-NETIF API გამოყენებით
  • TCP/IP-თან დაკავშირებული პარამეტრების (DHCP, IP და ა.შ.) მიღება და დაყენება
  • IP მოვლენების მიღება (დაკავშირება/გათიშვა)
  • აპლიკაციის სასიცოცხლო ციკლის კონტროლი (ინტერფეისის ზევით/ქვევით დაყენება)

B) საკომუნიკაციო დრაივერი, IO დრაივერი, მედია დრაივერი¶

კომუნიკაციის დრაივერი ESP-NETIF-თან მიმართებაში ამ ორ მნიშვნელოვან როლს ასრულებს:

  1. მოვლენების დამმუშავებლები: განსაზღვრეთ ESP-NETIF-თან ურთიერთქმედების ქცევითი ნიმუშები (მაგალითად: Ethernet link-up -&gt; netif-ის ჩართვა)
  2. წებოვანი IO ფენა: ადაპტირებს შეყვანის/გამოყვანის ფუნქციებს ESP-NETIF გადაცემის, მიღების და მიღების ბუფერის გასათავისუფლებლად.
  • driver_transmit-ის ინსტალაცია შესაბამის ESP-NETIF ობიექტზე, ისე, რომ ქსელის დასტიდან გამავალი პაკეტები გადაეცემა IO დრაივერს.
  • ზარები esp_netif_receive() შემომავალი მონაცემების ქსელის სტეკში გადასაცემად

გ) ESP-NETIF, ყოფილი tcpip_adapter¶

ESP-NETIF არის შუამავალი IO დრაივერსა და ქსელურ დასტას შორის, რომელიც აკავშირებს პაკეტის მონაცემთა გზას ამ ორს შორის. ამრიგად, ის უზრუნველყოფს ინტერფეისების ერთობლიობას დრაივერის ESP-NETIF ობიექტზე მისაერთებლად (გაშვების დრო) და ქსელური დასტის კონფიგურაციისთვის ( კომპილაცია დრო). გარდა ამისა, API ის ერთობლიობა მოცემულია ქსელური ინტერფეისის სასიცოცხლო ციკლის და მისი TCP/IP თვისებების გასაკონტროლებლად. ზოგადი შეჯამებისთვის, ESP-NETIF საჯარო ინტერფეისი შეიძლება დაიყოს შემდეგ 6 ჯგუფად:

  1. ინიციალიზაციის API s (ESP-NETIF ეგზემპლარის შესაქმნელად და კონფიგურაციისთვის)
  2. შეყვანა/გამოყვანა API (მონაცემების გადასაცემად IO დრაივერსა და ქსელის სტეკს შორის)
  3. მოვლენა ან მოქმედება API
  • გამოიყენება ქსელის ინტერფეისის სასიცოცხლო ციკლის მართვისთვის
  • ESP-NETIF უზრუნველყოფს საშენ მასალას მოვლენების დამმუშავებლების შესაქმნელად.
  1. ქსელის ინტერფეისის ძირითადი თვისებების დამყენებლები და მიმღებები
  2. ქსელის სტეკის აბსტრაქცია: მომხმარებლის TCP/IP სტეკთან ურთიერთქმედების უზრუნველყოფა
  • ინტერფეისის აწევა ან დაწევა
  • DHCP სერვერი და კლიენტი API
  • DNS API
  1. დრაივერის კონვერტაციის კომუნალური პროგრამები

დ) ქსელის სტეკი¶

ქსელურ დასტას არ აქვს საჯარო ურთიერთქმედება აპლიკაციის კოდთან საჯარო ინტერფეისებთან მიმართებაში და სრულად უნდა იყოს აბსტრაგირებული ESP-NETIF API .

ESP-NETIF პროგრამისტის სახელმძღვანელო¶

ნაგულისხმევი ინტერფეისების ძირითადი ინიციალიზაციისთვის იხილეთ მაგალითის განყოფილება:

  • WiFi სადგური: wifi/დაწყება/სადგური/ძირითადი/სადგურის_მაგალითი_ძირითადი.c
  • WiFi წვდომის წერტილი: wifi/getting_started/softAP/main/softap_example_main.c
  • ეთერნეტი: ethernet/basic/main/ethernet_example_main.c

უფრო კონკრეტული შემთხვევებისთვის, გთხოვთ, იხილოთ ეს სახელმძღვანელო: ESP-NETIF-ის მორგებული შეყვანის/გამოყვანის დრაივერი.

WiFi-ის ნაგულისხმევი ინიციალიზაცია¶

ინიციალიზაციის კოდი, ისევე როგორც ნაგულისხმევი ინტერფეისების, როგორიცაა softAP და station, რეგისტრაციის მოვლენების დამმუშავებლები, მოცემულია ორ ცალკეულ API 8, რათა ხელი შეუწყოს აპლიკაციების უმეტესობისთვის გაშვების კოდის გამარტივებას:

  • esp_netif_create_default_wifi_ap()
  • esp_netif_create_default_wifi_sta()

გთხოვთ გაითვალისწინოთ, რომ ეს ფუნქციები აბრუნებენ esp_netif სახელური, ანუ ქსელის ინტერფეისის ობიექტის მაჩვენებელი, რომელიც გამოყოფილი და კონფიგურირებულია ნაგულისხმევი პარამეტრებით, რაც, შესაბამისად, ნიშნავს, რომ:

  • შექმნილი ობიექტი უნდა განადგურდეს, თუ აპლიკაცია უზრუნველყოფს ქსელის დეინიციალიზაციას.
  • ესენი ნაგულისხმევი ინტერფეისები არ უნდა შეიქმნას რამდენჯერმე, თუ შექმნილი სახელური არ წაიშლება გამოყენებით esp_netif_destroy().

თუ IP მისამართის დაკარგვის ტაიმერი ამოიწურა (რაც ნიშნავს, რომ ინტერფეისი კონფიგურირებულ ინტერვალზე მეტხანს იყო გათიშული), მაშინ ძველი IP ინფორმაცია ნულის ტოლი იქნება.

  • ESP_ERR_ESP_NETIF_INVALID_PARAMS

  • ip_infoწარმატების შემთხვევაში, IP ინფორმაცია დაბრუნდება ამ არგუმენტში.

esp_err_t esp_netif_set_ip_info(esp_netif_t *esp_netif, კონსტ esp_netif_ip_info_t *ip_info)¶

ინტერფეისის IP მისამართის ინფორმაციის დაყენება.

ეს ფუნქცია ძირითადად გამოიყენება ინტერფეისზე სტატიკური IP მისამართის დასაყენებლად.

თუ ინტერფეისი ჩართულია, ახალი IP ინფორმაცია პირდაპირ TCP/IP დასტაში დაყენდება.

ESP-NETIF ინსტანციაში შენახული IP ინფორმაციის ასლი ასევე განახლდება (ეს ასლი ბრუნდება, თუ IP მისამართის მოთხოვნა ხდება ინტერფეისის გათიშვის დროს).

შენიშვნა
DHCP client/server must be stopped (if enabled for this interface) before setting new IP information.
შენიშვნა
Calling this interface for may generate a SYSTEM_EVENT_STA_GOT_IP or SYSTEM_EVENT_ETH_GOT_IP event.
დაბრუნება
  • ESP_OK
  • ESP_ERR_ESP_NETIF_INVALID_PARAMS
  • ESP_ERR_ESP_NETIF_DHCP_NOT_STOPPED თუ DHCP სერვერი ან კლიენტი კვლავ მუშაობს
პარამეტრები
  • esp_netif: esp-netif ინსტანციის სახელური
  • ip_infoმითითებულ ინტერფეისზე დასაყენებელი IP ინფორმაცია

esp_err_t esp_netif_set_old_ip_info(esp_netif_t *esp_netif, კონსტ esp_netif_ip_info_t *ip_info)¶

ინტერფეისის ძველი IP ინფორმაციის დაყენება.

ეს ფუნქცია გამოიძახება DHCP კლიენტიდან (თუ ჩართულია), ახალი IP მისამართის დაყენებამდე. ის ასევე გამოიძახება SYSTEM_EVENT_STA_CONNECTED და SYSTEM_EVENT_ETH_CONNECTED მოვლენების ნაგულისხმევი დამმუშავებლებისგან.

ამ ფუნქციის გამოძახება ინახავს ადრე კონფიგურირებულ IP მისამართს, რომლის გამოყენებაც შესაძლებელია იმის დასადგენად, შეიცვლება თუ არა IP მისამართი მომავალში.

თუ ინტერფეისი გათიშულია ან დიდი ხნის განმავლობაში არ მუშაობს, „IP მისამართის დაკარგვის ტაიმერი“ ამოიწურება (კონფიგურირებული ინტერვალის შემდეგ) და ძველი IP ინფორმაცია ნულზე დაყენდება.

esp_err_t esp_netif_set_dns_info(esp_netif_t *esp_netif, esp_netif_dns_type_t ტიპი, esp_netif_dns_info_t *დნმ)¶

DNS სერვერის ინფორმაციის დაყენება.

ეს ფუნქცია განსხვავებულად იქცევა, თუ DHCP სერვერი ან კლიენტი ჩართულია.

თუ DHCP კლიენტი ჩართულია, ძირითადი და სარეზერვო DNS სერვერები ავტომატურად განახლდება DHCP იჯარის ხელშეკრულებიდან, თუ დაყენებულია შესაბამისი DHCP პარამეტრები. სარეზერვო DNS სერვერი არასდროს განახლდება DHCP იჯარის ხელშეკრულებიდან და შექმნილია ამ API მეშვეობით დასაყენებლად. თუ DHCP კლიენტი გამორთულია, ყველა DNS სერვერის ტიპის დაყენება შესაძლებელია მხოლოდ ამ API მეშვეობით.

თუ DHCP სერვერი ჩართულია, DHCP სერვერი გამოიყენებს მთავარი DNS სერვერის პარამეტრს, რათა DHCP კლიენტებს ( Wi-Fi სადგურები) მიაწოდოს DNS სერვერის ვარიანტი.

  • ნაგულისხმევი მთავარი DNS სერვერი, როგორც წესი, თავად Wi-Fi AP ინტერფეისის IP მისამართია.
  • ამ ფუნქციას შეუძლია მისი გადაფარვა სერვერის ტიპის ESP_NETIF_DNS_MAIN დაყენებით.
  • Wi-Fi AP ინტერფეისისთვის სხვა DNS სერვერის ტიპები არ არის მხარდაჭერილი.

დაბრუნება
  • ESP_OK წარმატების შემთხვევაში
  • ESP_ERR_ESP_NETIF_INVALID_PARAMS არასწორი პარამეტრები
პარამეტრები
  • esp_netif: esp-netif ინსტანციის სახელური
  • typeდასაყენებელი DNS სერვერის ტიპი: ESP_NETIF_DNS_MAIN, ESP_NETIF_DNS_BACKUP, ESP_NETIF_DNS_FALLBACK
  • dns: დასაყენებელი DNS სერვერის მისამართი

esp_err_t esp_netif_create_default_wifi_mesh_netifs(esp_netif_t **p_netif_sta, esp_netif_t **p_netif_ap)¶

ქმნის სტანდარტულ STA და AP ქსელურ ინტერფეისებს esp-mesh-ისთვის.

ორივე netif თითქმის იდენტურია ნაგულისხმევი სადგურისა და softAP-ისა, მაგრამ DHCP კლიენტი და სერვერი გამორთულია. გაითვალისწინეთ, რომ DHCP კლიენტი, როგორც წესი, ჩართულია მხოლოდ იმ შემთხვევაში, თუ მოწყობილობა გადაყვანილია root კვანძად.

აბრუნებს შექმნილ ინტერფეისებს, რომელთა იგნორირება შესაძლებელია პარამეტრების NULL-ზე დაყენებით, თუ აპლიკაციის კოდს არ სჭირდება ინტერფეისის ეგზემპლარების შენახვა შემდგომი დამუშავებისთვის.

დაბრუნება
ESP_OK on success
პარამეტრები
  • p_netif_sta: მაჩვენებელი, სადაც შენახულია შედეგად მიღებული STA ინტერფეისი (თუ არა NULL)
  • p_netif_ap: მაჩვენებელი, სადაც შენახულია შედეგად მიღებული AP ინტერფეისი (თუ არა NULL)

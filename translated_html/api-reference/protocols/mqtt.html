ESP-MQTT¶

მიმოხილვა¶

ESP-MQTT არის MQTT პროტოკოლის კლიენტის იმპლემენტაცია (MQTT არის მსუბუქი გამოქვეყნების/გამოწერის შეტყობინებების პროტოკოლი).

მახასიათებლები¶

  • მხარს უჭერს MQTT-ს TCP-ის საშუალებით, SSL-ს mbedtls-ით, MQTT-ს Websocket-ის საშუალებით, MQTT-ს Websocket Secure-ის საშუალებით.
  • მარტივი დაყენება URI-ით
  • მრავალი ეგზემპლარი (მრავალი კლიენტი ერთ აპლიკაციაში)
  • მხარდაჭერა გამოწერის, გამოქვეყნების, ავთენტიფიკაციის, ბოლო ანდერძის შეტყობინებების, შენახვის პინგების და QoS-ის 3 დონისთვის (ეს უნდა იყოს სრულად ფუნქციონალური კლიენტი).

განაცხადის მაგალითი¶

  • პროტოკოლები/mqtt/tcp: MQTT tcp-ის საშუალებით, ნაგულისხმევი პორტი 1883
  • პროტოკოლები/mqtt/ssl: MQTT tcp-ის მეშვეობით, ნაგულისხმევი პორტი 8883
  • პროტოკოლები/mqtt/ssl_pskMQTT tcp-ის მეშვეობით, ავტორიზაციისთვის წინასწარ გაზიარებული გასაღებების გამოყენებით, ნაგულისხმევი პორტი 8883
  • პროტოკოლები/mqtt/ws: MQTT Websocket-ის მეშვეობით, ნაგულისხმევი პორტი 80
  • პროტოკოლები/mqtt/wss: MQTT Websocket Secure-ის მეშვეობით, ნაგულისხმევი პორტი 443

კონფიგურაცია¶

URI¶

  • ამჟამად მხარდაჭერა mqtt, mqtts, ws, wss სქემები
  • MQTT TCP-ზე ნიმუშები:
    • mqtt://mqtt.eclipse.orgMQTT TCP-ის მეშვეობით, ნაგულისხმევი პორტი 1883:
    • mqtt://mqtt.eclipse.org:1884 MQTT TCP-ის საშუალებით, პორტი 1884:
    • mqtt://username:password@mqtt.eclipse.org:1884 MQTT TCP-ის საშუალებით, პორტი 1884, მომხმარებლის სახელით და პაროლით
  • MQTT SSL-ის მეშვეობით ნიმუშები:
    • mqtts://mqtt.eclipse.org: MQTT SSL-ის საშუალებით, პორტი 8883
    • mqtts://mqtt.eclipse.org:8884: MQTT SSL-ის საშუალებით, პორტი 8884
  • MQTT Websocket-ის ნიმუშების გამოყენებით:
    • ws://mqtt.eclipse.org:80/mqtt
  • MQTT Websocket Secure-ის მეშვეობით:
    • wss://mqtt.eclipse.org:443/mqtt
  • მინიმალური კონფიგურაციები:
const esp_mqtt_client_config_t mqtt_cfg = {
    .uri = "mqtt://mqtt.eclipse.org",
    // .user_context = (void *)your_context
};
esp_mqtt_client_handle_t client = esp_mqtt_client_init(&mqtt_cfg);
esp_mqtt_client_register_event(client, ESP_EVENT_ANY_ID, mqtt_event_handler, client);
esp_mqtt_client_start(client);
  • შენიშვნა: სტანდარტულად, mqtt კლიენტი იყენებს მოვლენების ციკლის ბიბლიოთეკას დაკავშირებული mqtt მოვლენების (დაკავშირებული, გამოწერილი, გამოქვეყნებული და ა.შ.) გამოსაქვეყნებლად.

SSL¶

  • მიიღეთ სერტიფიკატი სერვერიდან, მაგალითად: mqtt.eclipse.org openssl s_client -showcerts -connect mqtt.eclipse.org:8883 </dev/null 2>/dev/null|openssl x509 -outform PEM >mqtt_eclipse_org.pem
  • შეამოწმეთ ნიმუშის განაცხადი: examples/mqtt_ssl
  • კონფიგურაცია:
const esp_mqtt_client_config_t mqtt_cfg = {
    .uri = "mqtts://mqtt.eclipse.org:8883",
    .event_handle = mqtt_event_handler,
    .cert_pem = (const char *)mqtt_eclipse_org_pem_start,
};

თუ სერტიფიკატი არ არის null-შეწყვეტილი, მაშინ cert_len ასევე უნდა დაყენდეს. SSL-თან დაკავშირებული სხვა კონფიგურაციის პარამეტრებია:

  • use_global_ca_store: სერვერის სერტიფიკატის დასადასტურებლად გამოიყენეთ გლობალური სერტიფიკატების საცავი, იხილეთ esp-tls.h დამატებითი ინფორმაციისთვის
  • client_cert_pemSSL ორმხრივი ავთენტიფიკაციისთვის, სერტიფიკატის მონაცემებზე მითითება PEM ან DER ფორმატში, ნაგულისხმევი მნიშვნელობაა NULL, არ არის საჭირო, თუ ორმხრივი ავთენტიფიკაცია არ არის საჭირო.
  • client_cert_lenclient_cert_pem-ის მიერ მითითებული ბუფერის სიგრძე. შესაძლოა იყოს 0 ნულოვანი ტერმინალით დასრულებული pem-ისთვის.
  • client_key_pemSSL ორმხრივი ავთენტიფიკაციისთვის, კერძო გასაღების მონაცემებზე მითითება PEM ან DER ფორმატში, ნაგულისხმევი მნიშვნელობაა NULL, არ არის საჭირო, თუ ორმხრივი ავთენტიფიკაცია არ არის საჭირო.
  • client_key_lenclient_key_pem-ის მიერ მითითებული ბუფერის სიგრძე. null-დასრულებული pem-ისთვის შესაძლოა იყოს 0.
  • psk_hint_key: esp_tls.h-ში განსაზღვრული PSK სტრუქტურის მითითება PSK ავთენტიფიკაციის ჩასართავად (როგორც სერტიფიკატის ვერიფიკაციის ალტერნატივა). თუ არა NULL და სერვერის/კლიენტის სერტიფიკატები NULL-ია, PSK ჩართულია.
  • alpn_protosALPN-ისთვის გამოსაყენებელი პროტოკოლების NULL-ით დასრულებული სია.

უკანასკნელი ანდერძი და აღთქმა¶

MQTT საშუალებას იძლევა, რომ ბოლო ანდერძისა და აღთქმის (LWT) შეტყობინებით სხვა კლიენტებს აცნობოს, როდესაც კლიენტი უპატივცემულოდ წყვეტს კავშირს. ეს კონფიგურირებულია შემდეგი ველებით esp_mqtt_client_config_t-სტრუქტურა.

  • lwt_topic: მითითება LWT შეტყობინების თემაზე
  • lwt_msg: LWT შეტყობინების მითითება
  • lwt_msg_len: LWT შეტყობინების სიგრძე, აუცილებელია, თუ lwt_msg არ არის ნულით დასრულებული
  • lwt_qos: LWT შეტყობინების მომსახურების ხარისხი
  • lwt_retain: განსაზღვრავს LWT შეტყობინების შენარჩუნების ფლაგს

სხვა კონფიგურაციის პარამეტრები¶

  • disable_clean_session: განსაზღვრავს დაკავშირების შეტყობინებისთვის სუფთა სესიის ფლაგს, ნაგულისხმევად სუფთა სესიაზეა მითითებული.
  • keepalive: განსაზღვრავს, რამდენ წამს დაელოდება კლიენტი პინგ პასუხს გათიშვის წინ, ნაგულისხმევი მნიშვნელობაა 120 წამი.
  • disable_auto_reconnect: ჩართეთ კლიენტის სერვერთან ხელახლა დაკავშირების შეჩერება შეცდომების ან გათიშვის შემდეგ
  • user_context: მორგებული კონტექსტი, რომელიც გადაეცემა მოვლენის დამმუშავებელს
  • task_prioMQTT დავალების პრიორიტეტი, ნაგულისხმევად 5
  • task_stackMQTT დავალებების დასტის ზომა, ნაგულისხმევად 6144 ბაიტი, ამის დაყენება გადაფარავს menuconfig ის პარამეტრს.
  • buffer_size: MQTT გაგზავნის/მიღების ბუფერის ზომა, ნაგულისხმევი მნიშვნელობაა 1024 ბაიტი
  • username: მიუთითებს ბროკერთან დასაკავშირებლად გამოყენებულ მომხმარებლის სახელზე
  • password: ბროკერთან დასაკავშირებლად გამოყენებული პაროლის მითითება
  • client_id: კლიენტის ID-ზე მითითება, ნაგულისხმევად ESP32_%CHIPID% სადაც %CHIPID% არის MAC მისამართის ბოლო 3 ბაიტი თექვსმეტობით ფორმატში.
  • hostMQTT ბროკერის დომენი (ipv4 როგორც სტრიქონი), uri-ს დაყენება ამას გადაფარავს.
  • portMQTT ბროკერის პორტი, UR-ში პორტის მითითებით, ამას გადაფარავს.
  • transport: ადგენს ტრანსპორტის პროტოკოლს, uri-ს დაყენება ამას გადაფარავს
  • refresh_connection_after_ms: კავშირის განახლება ამ მნიშვნელობის შემდეგ (მილიწამებში)
  • event_handle: MQTT მოვლენების სახელური, როგორც უკუკავშირი მემკვიდრეობით რეჟიმში
  • event_loop_handle: MQTT მოვლენების ციკლის ბიბლიოთეკის სახელური

მეტი ვარიანტისთვის esp_mqtt_client_config_t, გთხოვთ, იხილოთ ქვემოთ მოცემული API მითითება

პარამეტრების შეცვლა პროექტის კონფიგურაციის მენიუში¶

MQTT-ის პარამეტრების პოვნა შეგიძლიათ გამოყენებით idf.py menuconfig, კომპონენტის კონფიგურაციის -&gt; ESP-MQTT კონფიგურაციის ქვეშ

შემდეგი პარამეტრები ხელმისაწვდომია:

  • CONFIG_MQTT_PROTOCOL_311: ჩართავს MQTT პროტოკოლის 3.1.1 ვერსიას
  • CONFIG_MQTT_TRANSPORT_SSL, CONFIG_MQTT_TRANSPORT_WEBSOCKET: ჩართავს MQTT-ის სპეციფიკურ სატრანსპორტო ფენას, როგორიცაა SSL, WEBSOCKET, WEBSOCKET_SECURE
  • CONFIG_MQTT_CUSTOM_OUTBOX: გამორთავს mqtt_outbox-ის ნაგულისხმევ იმპლემენტაციას, რათა კონკრეტული იმპლემენტაციის მიწოდება იყოს შესაძლებელი.

ღონისძიებები¶

MQTT კლიენტმა შეიძლება გამოაქვეყნოს შემდეგი მოვლენები:

  • MQTT_EVENT_BEFORE_CONNECTკლიენტი ინიციალიზებულია და ბროკერთან დაკავშირების დაწყებას აპირებს.
  • MQTT_EVENT_CONNECTEDკლიენტმა წარმატებით დაამყარა კავშირი ბროკერთან. კლიენტი ახლა მზადაა მონაცემების გასაგზავნად და მისაღებად.
  • MQTT_EVENT_DISCONNECTEDკლიენტმა შეწყვიტა კავშირი მონაცემების წაკითხვის ან ჩაწერის შეუძლებლობის გამო, მაგ. სერვერის მიუწვდომლობის გამო.
  • MQTT_EVENT_SUBSCRIBEDბროკერმა დაადასტურა კლიენტის გამოწერის მოთხოვნა. მოვლენის მონაცემები შეიცავს გამოწერის შეტყობინების შეტყობინების ID-ს.
  • MQTT_EVENT_UNSUBSCRIBEDბროკერმა დაადასტურა კლიენტის გამოწერის გაუქმების მოთხოვნა. მოვლენის მონაცემები შეიცავს გამოწერის გაუქმების შეტყობინების ID-ს.
  • MQTT_EVENT_PUBLISHEDბროკერმა დაადასტურა კლიენტის გამოქვეყნებული შეტყობინება. ეს გამოქვეყნდება მხოლოდ მომსახურების ხარისხის 1 და 2 დონისთვის, რადგან 0 დონე არ იყენებს დადასტურებებს. მოვლენის მონაცემები შეიცავს გამოქვეყნებული შეტყობინების შეტყობინების ID-ს.
  • MQTT_EVENT_DATAკლიენტმა მიიღო გამოქვეყნების შეტყობინება. მოვლენის მონაცემები შეიცავს: შეტყობინების ID-ს, გამოქვეყნებული თემის სახელს, მიღებულ მონაცემებს და მის სიგრძეს. მონაცემებისთვის, რომლებიც აღემატება შიდა ბუფერის ჯერადს MQTT_EVENT_DATA გამოქვეყნდება და current_data_offset და total_data_len ფრაგმენტული შეტყობინების თვალყურის დევნების მიზნით განახლებული მოვლენების მონაცემებიდან.
  • MQTT_EVENT_ERRORკლიენტს შეცდომა შეექმნა. esp_mqtt_error_type_t -დან error_handle იმ შემთხვევაში, მონაცემების გამოყენება შესაძლებელია შეცდომის ტიპის შემდგომი დასადგენად. შეცდომის ტიპი განსაზღვრავს, თუ რომელი ნაწილები error_handle სტრუქტურა შევსებულია.

სტრუქტურა esp_mqtt_error_codes¶

MQTT შეცდომის კოდის სტრუქტურა, რომელიც უნდა გადაეცეს კონტექსტური ინფორმაციის სახით ERROR მოვლენაში.

მნიშვნელოვანია: ეს სტრუქტურა ვრცელდება esp_tls_last_error შეცდომის სტრუქტურა და უკუთავსებადია მასთან (ამიტომ შეიძლება დაქვეითდეს და განიხილებოდეს, როგორც esp_tls_last_error შეცდომაა, მაგრამ რეკომენდებულია აპლიკაციების განახლება, თუ ადრე ამ გზით გამოიყენებოდა)

გამოიყენეთ ეს სტრუქტურა პირდაპირ, ჯერ error_type-ის შემოწმებით და შემდეგ შესაბამისი შეცდომის კოდის შემოწმებით, შეცდომის წყაროდან გამომდინარე:

| შეცდომის_ტიპი | დაკავშირებული წევრის ცვლადები | შენიშვნა | | MQTT_ERROR_TYPE_ESP_TLS | esp_tls_last_esp_err, esp_tls_stack_err, esp_tls_cert_verify_flags | შეცდომა დაფიქსირებულია esp-tls-დან | | MQTT_ERROR_TYPE_CONNECTION_REFUSED | connect_return_code | შიდა შეცდომა დაფიქსირებულია MQTT ბროკერის მიერ კავშირის დროს |

MQTT_EVENT_DATA¶

მონაცემთა მოვლენა, დამატებითი კონტექსტი:

  • msg_id შეტყობინების ID
  • თემის მითითება მიღებულ თემაზე
  • topic_len თემის სიგრძე
  • მიღებულ მონაცემებზე მონაცემების მაჩვენებელი
  • ამ მოვლენის მონაცემების სიგრძე data_len
  • ამ მოვლენის მიმდინარე მონაცემების current_data_offset ოფსეტი
  • total_data_len მიღებული მონაცემების საერთო სიგრძე შენიშვნა: ერთი შეტყობინებისთვის შესაძლებელია მრავალი MQTT_EVENT_DATA-ს გაშვება, თუ ის შიდა ბუფერზე გრძელია. ამ შემთხვევაში, მხოლოდ პირველი მოვლენა შეიცავს თემის მაჩვენებელს და სიგრძეს, დანარჩენი კი შეიცავს მონაცემებს მხოლოდ მიმდინარე მონაცემების სიგრძით და მიმდინარე მონაცემთა ოფსეტის განახლებით.

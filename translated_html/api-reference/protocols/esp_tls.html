ESP-TLS¶

მიმოხილვა¶

ESP-TLS კომპონენტი უზრუნველყოფს გამარტივებულ API ინტერფეისს ხშირად გამოყენებულ TLS ფუნქციონალზე წვდომისთვის. ის მხარს უჭერს ისეთ ხშირად გამოყენებულ სცენარებს, როგორიცაა CA სერტიფიცირების ვალიდაცია, SNI, ALPN მოლაპარაკება, არაბლოკირებადი კავშირი და სხვა. ყველა კონფიგურაციის მითითება შესაძლებელია... esp_tls_cfg_t მონაცემთა სტრუქტურა. დასრულების შემდეგ, TLS კომუნიკაცია შეიძლება განხორციელდეს შემდეგი API გამოყენებით:

  • esp_tls_conn_new(): ახალი TLS კავშირის გასახსნელად.
  • esp_tls_conn_read(): კავშირიდან წასაკითხად.
  • esp_tls_conn_write(): კავშირში ჩასაწერად.
  • esp_tls_conn_delete(): კავშირის გასათავისუფლებლად.

ნებისმიერი აპლიკაციის დონის პროტოკოლი, როგორიცაა HTTP1, HTTP2 და ა.შ., შეიძლება შესრულდეს ამ დონის თავზე.

განაცხადის მაგალითი¶

მარტივი HTTPS მაგალითი, რომელიც იყენებს ESP-TLS-ს უსაფრთხო სოკეტის კავშირის დასამყარებლად: პროტოკოლები/https_მოთხოვნა.

ESP-TLS კომპონენტი ის ხის სტრუქტურა¶

├── esp_tls.c
├── esp_tls.h
├── esp_tls_mbedtls.c
├── esp_tls_wolfssl.c
└── private_include
    ├── esp_tls_mbedtls.h
    └── esp_tls_wolfssl.h

ESP-TLS კომპონენტი აქვს ფაილი esp-tls/esp_tls.h რომლებიც შეიცავს კომპონენტი საჯარო API სათაურებს. შინაგანად ESP-TLS კომპონენტი თავისი ფუნქციონირებისთვის იყენებს mbedtls-სა და wolfssl-ს შორის არსებულ ორ SSL/TLS ბიბლიოთეკას შორის ერთ-ერთს. mbedtls-ისთვის სპეციფიკური API წარმოდგენილია esp-tls/private_include/esp_tls_mbedtls.h და wolfssl-ისთვის დამახასიათებელი API წარმოდგენილია esp-tls/private_include/esp_tls_wolfssl.h.

SSL/TLS ბიბლიოთეკის ძირითადი პარამეტრები¶

ESP-TLS კომპონენტი აქვს mbedtls-ის ან wolfssl-ის გამოყენების ვარიანტი, როგორც მათი ძირითადი SSL/TLS ბიბლიოთეკა. ნაგულისხმევად მხოლოდ mbedtls არის ხელმისაწვდომი და გამოიყენება, wolfssl SSL/TLS ბიბლიოთეკა საჯაროდ ხელმისაწვდომია შემდეგ მისამართზე: https://github.com/espressif/esp-wolfsslსაცავი გთავაზობთ wolfssl კომპონენტი ორობით ფორმატში, ასევე გთავაზობთ რამდენიმე მაგალითს, რომლებიც სასარგებლოა API გასაგებად. ლიცენზირებისა და სხვა ვარიანტების შესახებ ინფორმაციისთვის იხილეთ README.md საცავი. გთხოვთ, იხილოთ ქვემოთ მოცემული ვარიანტი თქვენს პროექტში wolfssl-ის გამოყენებისთვის.

შენიშვნა

As the library options are internal to ESP-TLS, switching the libries will not change ESP-TLS specific code for a project.

როგორ გამოვიყენოთ wolfssl ESP-IDF თან ერთად¶

თქვენს პროექტში wolfssl-ის გამოყენების ორი გზა არსებობს

  1. თქვენს პროექტში პირდაპირ დაამატეთ wolfssl როგორც კომპონენტი შემდეგი სამი ბრძანების გამოყენებით:

    (First change directory (cd) to your project directory)
mkdir components
cd components
git clone https://github.com/espressif/esp-wolfssl.git

  2. დაამატეთ wolfssl, როგორც დამატებითი კომპონენტი თქვენს პროექტში.

  • ჩამოტვირთეთ wolfssl შემდეგი ინსტრუქციით:

    git clone https://github.com/espressif/esp-wolfssl.git

  • ჩართეთ esp-wolfssl ESP-IDF ში თქვენი პროექტის CMakeLists.txt/Makefile-ში EXTRA_COMPONENT_DIRS-ის დაყენებით, როგორც შესრულებულია. wolfssl/მაგალითები. მითითებისთვის იხილეთ პროექტის არჩევითი ცვლადები აშენების სისტემა.

ზემოთ აღნიშნული ნაბიჯების შემდეგ, თქვენი პროექტის კონფიგურაციის მენიუში გექნებათ შესაძლებლობა აირჩიოთ wolfssl, როგორც ძირითადი SSL/TLS ბიბლიოთეკა, შემდეგნაირად:

idf.py/make menuconfig -> ESP-TLS -> choose SSL/TLS Library -> mbedtls/wolfssl

mbedtls-სა და wolfssl-ს შორის შედარება¶

ქვემოთ მოცემულ ცხრილში მოცემულია wolfssl-სა და mbedtls-ს შორის ტიპური შედარება, როდესაც პროტოკოლები/https_მოთხოვნა მაგალითი (which has server authentication) გაშვებული იყო როგორც SSL/TLS ბიბლიოთეკებით და ყველა შესაბამისი კონფიგურაციით ნაგულისხმევ პარამეტრებზე დაყენებული. (mbedtls IN_CONTENT length and OUT_CONTENT length were set to 16384 bytes and 4096 bytes respectively)

ქონება ვოლფსლი მბედტლსი
სულ მოხმარებული გროვა ~19 კბ ~37 კბ
გამოყენებული დავალებების დასტა ~2.2 კბ ~3.6 კბ
ურნის ზომა ~858 კბ ~736 კბ

შენიშვნა

These values are subject to change with change in configuration options and version of respective libraries.

შექმენით ახალი დაბლოკვის TLS/SSL კავშირი.

ეს ფუნქცია ბლოკირების მეთოდით ამყარებს TLS/SSL კავშირს მითითებულ ჰოსტთან.

შენიშვნა: ეს API არსებობს უკუთავსებადობის მიზნით. იგივე ფუნქციონალურობით ალტერნატიული ფუნქციაა esp_tls_conn_new_sync (და მისი ასინქრონული ვერსია esp_tls_conn_new_async)

დაბრუნება
pointer to esp_tls_t, or NULL if connection couldn’t be opened.
პარამეტრები
  • hostname: მასპინძლის ჰოსტის სახელი.
  • hostlen: ჰოსტის სახელის სიგრძე.
  • port: მასპინძლის პორტის ნომერი.
  • cfg: TLS კონფიგურაცია, როგორც esp_tls_cfg_t. თუ გსურთ არა-TLS კავშირის გახსნა, დატოვეთ ეს NULL. TLS კავშირისთვის, esp_tls_cfg_t-ზე გადასვლის მაჩვენებელი. მინიმუმ, ეს სტრუქტურა უნდა იყოს ნულოვანი ინიციალიზებული.

int esp_tls_conn_new_sync(კონსტ char *ჰოსტის სახელი, int ჰოსტლენი, int პორტი, კონსტ esp_tls_cfg_t *cfg, esp_tls_t *ტლს)¶

  • tlsesp-tls-ის მიმთითებელი, როგორც esp-tls-ის სახელური.

  • 0 თუ კავშირის დამყარება მიმდინარეობს.
  • 1 თუ კავშირი წარმატებით დამყარდა.
პარამეტრები
  • hostname: მასპინძლის ჰოსტის სახელი.
  • hostlen: ჰოსტის სახელის სიგრძე.
  • port: მასპინძლის პორტის ნომერი.
  • cfgTLS კონფიგურაცია, როგორც esp_tls_cfg_t. non_block ამ სტრუქტურის წევრი უნდა იყოს დაყენებული true-ზე.
  • tls: esp-tls-ის მაჩვენებელი, როგორც esp-tls-ის სახელური.

esp_err_t esp_tls_init_global_ca_store(void)¶

ეს ფუნქცია უნდა გამოიძახოს, თუ აპლიკაციას სურს ერთი და იგივე CA საცავის გამოყენება მრავალი კავშირისთვის. ეს ფუნქცია ინიციალიზაციას უკეთებს გლობალურ CA საცავის, რომლის დაყენებაც შემდეგ შესაძლებელია esp_tls_set_global_ca_store() ფუნქციის გამოძახებით. ეფექტურობისთვის, ეს ფუნქცია უნდა გამოიძახოს esp_tls_set_global_ca_store() ფუნქციის გამოძახებამდე.

დაბრუნება

  • სერტიფიკატის ორგანოს სერტიფიკატი შეიძლება იყოს სერტიფიკატების ჯაჭვი PEM ფორმატის შემთხვევაში, მაგრამ შეიძლება იყოს მხოლოდ ერთი სერტიფიკატი DER ფორმატის შემთხვევაში.
  • სერტიფიკატების ცვლადების სახელები და კერძო გასაღების ბუფერები და ზომები განისაზღვრება, როგორც გაერთიანებები, რომლებიც უზრუნველყოფენ უკუთავსებადობას მემკვიდრეობით მიღებული *_pem_buf და *_pem_bytes სახელებისთვის, რაც იმაზე მიუთითებს, რომ მხოლოდ PEM ფორმატი იყო მხარდაჭერილი. რეკომენდებულია ისეთი ზოგადი სახელების გამოყენება, როგორიცაა cacert_buf და cacert_bytes.
